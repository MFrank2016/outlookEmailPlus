## 1. 文档目标

本文档只讨论“临时邮箱能力”的测试设计，不重复展开业务需求和实现方案。

本 TDD 重点回答以下问题：

1. 这次改造需要测哪些层。
2. 哪些能力必须先写测试，哪些可以后补。
3. 用户侧、任务侧、兼容侧分别怎么测。
4. 如何在不依赖真实上游临时邮箱服务的前提下完成高可信验证。
5. 如何保证改造临时邮箱后，不误伤现有普通邮箱 external 链路。

## 2. 测试目标

本次测试的核心目标不是“证明新接口能通”，而是证明以下四件事同时成立：

1. 用户侧 `/api/temp-emails/*` 在切换到底层自建临时邮箱后仍然可用。
2. 任务侧只新增 `apply` / `finish` 两个接口后，仍可复用现有 `/api/external/messages*` 和提取接口完成注册流程。
3. 动态任务邮箱的授权规则正确，不能绕过 `consumer_key` 归属校验。
4. 普通 Outlook / IMAP 邮箱与原有验证码提取链路不被本次改造破坏。

## 3. 测试原则

### 3.1 测试优先级

按风险高低排序，本次测试优先级如下：

1. `mailbox_resolver` 分流与授权
2. external 任务邮箱读信兼容
3. 用户侧临时邮箱 API
4. temp mail provider / service 行为
5. 前端契约与最小冒烟

### 3.2 测试策略

本次遵循“单元为主，接口集成为核心，人工冒烟兜底”的测试策略。

建议比例：

1. 单元测试约 60% 到 70%
2. 集成 / 接口测试约 25% 到 35%
3. 前端契约 / 人工冒烟约 5% 到 10%

### 3.3 明确不做

本期不做以下高成本测试：

1. 真实上游临时邮箱服务端到端联调自动化
2. 真实 Cloudflare 域名配置自动化验证
3. 浏览器级完整 E2E 自动化
4. 压测级性能测试

原因不是不重要，而是本期核心风险在“服务分层、授权和兼容性”，这些风险更适合用单元和接口集成测试覆盖。

## 4. 测试分层设计

## 4.1 Provider / Adapter 单元测试

目标：验证临时邮箱上游适配层的输入输出转换、错误映射和容错行为。

必须覆盖：

1. `get_options()` 返回域名列表、默认域名和前缀规则
2. `generate_mailbox(prefix, domain)` 成功生成邮箱
3. `generate_mailbox()` 在指定域名不可用时返回 `DOMAIN_NOT_AVAILABLE`
4. `generate_mailbox()` 在前缀非法或冲突时返回对应错误
5. `list_messages(email_addr)` 返回统一消息结构
6. `get_message_detail(email_addr, message_id)` 返回详情结构
7. 上游超时、401、5xx、格式错误时映射为稳定错误

建议形式：

1. 纯单元测试
2. 对上游 HTTP 调用全部使用 mock
3. 不接真实网络

建议新增测试文件：

1. `tests/test_temp_mail_provider_custom.py`
2. `tests/test_temp_mail_service.py`

参考现有模式：

1. `tests/test_gptmail_error_handling.py`

## 4.2 Mailbox Resolver 单元测试

目标：验证 external 读信链路在普通邮箱和临时邮箱之间正确分流。

必须覆盖：

1. `resolve_mailbox(email_addr)` 能识别普通 `accounts` 邮箱
2. `resolve_mailbox(email_addr)` 能识别 `temp_emails` 中的用户邮箱
3. `resolve_mailbox(email_addr)` 能识别 `temp_emails` 中的任务邮箱
4. 不存在的邮箱返回明确未找到
5. 普通邮箱走 `allowed_emails` 范围校验
6. 任务邮箱走 `consumer_key` 归属校验
7. `finished` 任务邮箱被拒绝读取
8. 任务邮箱不能被其他 API consumer 读取

这一层是本次最关键的保护网，因为它决定 `/api/external/messages*` 是否会误伤现有链路。

建议新增测试文件：

1. `tests/test_mailbox_resolver.py`

## 4.3 Repository / 数据兼容测试

目标：验证新增字段和消息缓存行为不会破坏已有数据读写。

必须覆盖：

1. `temp_emails` 新增字段写入与读取
2. `visible_in_ui` 对用户邮箱与任务邮箱的区分
3. `task_token` 唯一性
4. `consumer_key`、`caller_id`、`task_id` 写入成功
5. `finished_at` 回写成功
6. `temp_email_messages` 以 `(email_address, message_id)` 维度保存，避免不同邮箱消息 ID 冲突
7. 旧缓存详情不被简化 payload 覆盖

参考现有模式：

1. `tests/test_temp_email_image_support.py`

说明：

这层不需要做复杂 migration 自动化测试，但至少要验证目标仓储写法符合新模型要求。

## 4.4 用户侧 API 集成测试

目标：验证 Web 登录态下的临时邮箱接口行为符合目标契约。

必须覆盖接口：

1. `GET /api/temp-emails/options`
2. `GET /api/temp-emails`
3. `POST /api/temp-emails/generate`
4. `GET /api/temp-emails/{email_addr}/messages`
5. `GET /api/temp-emails/{email_addr}/messages/{message_id}`
6. `GET /api/temp-emails/{email_addr}/extract-verification`
7. `DELETE /api/temp-emails/{email_addr}/messages/{message_id}`
8. `DELETE /api/temp-emails/{email_addr}/clear`
9. `DELETE /api/temp-emails/{email_addr}`
10. `POST /api/temp-emails/{email_addr}/refresh`

必须覆盖场景：

1. 已登录用户可获取 `options`
2. `options` 返回多域名时带 `auto_or_manual` 策略
3. 仅返回用户手动创建的 `visible_in_ui=true` 邮箱
4. 任务邮箱不会出现在用户列表
5. 创建邮箱支持空前缀、指定前缀、指定域名、自动分配域名
6. 指定不可用域名返回 400
7. 提取接口统一返回 `verification_code` 和 `verification_link`
8. 读取邮件列表和详情时使用 temp mail 服务，不走普通邮箱链路
9. 删除邮件、清空邮箱、删除邮箱后列表状态正确
10. 未登录时返回登录失败或鉴权失败

建议新增测试文件：

1. `tests/test_temp_emails_api.py`

说明：

`tests/test_api_extract_verification.py` 当前是占位测试，不建议继续扩写，应以新的 temp-email 路径重新建立有效集成测试。

## 4.5 任务侧 API 集成测试

目标：验证 `apply` / `finish` 两个最小新增接口正确落地。

必须覆盖接口：

1. `POST /api/external/temp-emails/apply`
2. `POST /api/external/temp-emails/{task_token}/finish`

必须覆盖场景：

1. 缺失 `X-API-Key` 返回 401
2. API Key 无效返回 401 或 403
3. 缺失 `caller_id` / `task_id` 返回 400
4. 成功申请邮箱时返回 `email`、`task_token`、`visible_in_ui=false`
5. 指定前缀和域名时返回符合条件的邮箱
6. 自动分配域名时返回系统选中的域名
7. 任务邮箱成功落库且归属当前 consumer
8. `finish` 成功后状态变为 `finished`
9. 重复 `finish` 返回 `TASK_ALREADY_FINISHED`
10. 使用其他 consumer 的 API Key 调用 `finish` 被拒绝
11. `finish` 后新发起的 `wait-message` 返回 `TASK_FINISHED`
12. `finish` 后 probe 查询返回 `cancelled` 或等价稳定终态

建议新增测试文件：

1. `tests/test_external_temp_emails_api.py`

参考现有模式：

1. `tests/test_external_api.py`

## 4.6 External 读信兼容集成测试

目标：验证“最小新增接口 + 复用现有 external 读信接口”的核心方案成立。

这是本次改造最重要的业务闭环测试。

必须覆盖场景：

1. `apply` 成功后，`GET /api/external/messages?email=...` 可读取任务邮箱消息
2. `GET /api/external/messages/latest?email=...` 可读取任务邮箱最新邮件
3. `GET /api/external/verification-code?email=...` 可提取验证码
4. `GET /api/external/verification-link?email=...` 可提取验证链接
5. `GET /api/external/wait-message?email=...` 对任务邮箱仍可工作
6. 任务邮箱 `finish` 后，再调用上述读取接口会被拒绝
7. 其他 consumer 读取该任务邮箱时返回 `FORBIDDEN`
8. 普通 Outlook / IMAP 邮箱调用同一组接口仍然通过
9. `finish` 发生时，已创建 probe 不会继续挂到普通 timeout 语义

建议新增测试文件：

1. `tests/test_external_api_temp_mail_compat.py`

参考现有模式：

1. `tests/test_external_api.py`

## 4.7 前端契约测试

目标：验证前端在最小范围内正确接入新接口，不再依赖旧的 GPTMail 逻辑。

必须覆盖：

1. 临时邮箱页面使用 `/api/temp-emails/options`
2. 创建邮箱时会提交 `prefix` 和 `domain`
3. 提取入口调用 `/api/temp-emails/{email_addr}/extract-verification`
4. 列表只渲染用户邮箱，不混入任务邮箱
5. 前端展示文案不再包含正式产品能力上的 `GPTMail`
6. 共享函数 `copyVerificationInfo` 或等价调用链支持按邮箱类型分流
7. temp-email 页面创建前会先读取 `/api/temp-emails/options`

建议方式：

1. 对现有 JS 做文本契约测试
2. 不强行引入新的前端测试框架

可复用参考：

1. `tests/test_temp_email_image_support.py` 中的 `TempEmailImageFrontendContractTests`

## 4.8 人工冒烟测试

## 11. 完成标准

本 TDD 对应的测试完成标准如下：

1. 新增单元测试能够覆盖临时邮箱 provider、service、resolver 三个核心层。
2. 用户侧新增和调整的 `/api/temp-emails/*` 接口都有有效集成测试。
3. 任务侧 `apply` / `finish` 有有效集成测试。
4. external 兼容链路至少存在一条完整闭环测试：`apply -> read -> extract -> finish`。
5. 现有普通邮箱 external 读信测试全部保持通过。
6. 不依赖真实上游服务即可稳定执行。

## 12. 当前结论

1. 本次最重要的测试不是 UI 细节，而是 `mailbox_resolver`、动态授权和 external 兼容链路。
2. 任务邮箱能力是否成立，关键不在新增多少接口，而在“申请后能否继续复用 existing external 读信接口”。
3. 用户侧 temp email 测试需要从旧的 `GPTMail` 路径思维中抽离出来，围绕新的 `/api/temp-emails/*` 契约重新建立。
4. 回归测试必须覆盖普通邮箱链路，否则这次改造风险不可控。
