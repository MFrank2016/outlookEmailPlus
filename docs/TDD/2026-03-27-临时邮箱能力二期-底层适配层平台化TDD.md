## 1. 文档目标

本文档只讨论二期 temp-mail 平台化改造的测试设计，不重复展开业务需求和实现细节。

本 TDD 重点回答以下问题：

1. 二期“真实接入实现”的测试验收标准是什么。
2. 哪些测试属于产品层保护，哪些属于底层适配层保护。
3. 为什么本期要把测试重心从“页面能不能点通”转到“适配层是否真正成型”。
4. 如何在不依赖真实外部 provider 联调的前提下，建立高可信测试体系。
5. 如何保证后续替换底层来源时，上层 `/api/temp-emails/*` 和 `/api/external/*` 契约不回归。

## 2. 二期测试目标

二期测试不是为了证明“再多一个接口能跑通”，而是为了证明以下五件事同时成立：

1. temp-mail 的上层产品契约保持稳定，用户侧和任务侧路径不变。
2. 底层临时邮箱来源已经被收进内部 provider 适配层，而不是继续泄漏到 controller / external / 前端。
3. mailbox 在本项目中已成为正式资源对象，而不只是一个 `email_addr` 字符串。
4. 更换底层来源时，只需要调整 provider 相关实现和配置，而不需要重写上层产品 API。
5. Outlook / IMAP 普通邮箱 external 能力不被本次平台化改造误伤。

## 3. “真实接入实现”的测试定义

本期测试的总验收语句定义如下：

当底层临时邮箱来源切换时，用户侧 `/api/temp-emails/*`、任务侧 `/api/external/*`、统一提取语义和前端页面交互均保持不变，且新来源只通过内部 provider 适配层接入并由 mailbox 内部模型承载其操作句柄时，才视为真实接入实现。

据此，二期测试必须覆盖四个关键词：

1. 上层路径稳定
2. provider 隔离成立
3. mailbox model 成立
4. 底层替换不反向污染上层

## 4. 测试原则

### 4.1 优先级原则

本期测试优先级按风险高低排序如下：

1. mailbox internal model 与 provider contract
2. temp mail service 与 provider factory
3. external 兼容回归
4. settings/runtime 语义一致性
5. 用户侧 temp-mail API 稳定性
6. 前端契约与最小冒烟

### 4.2 测试策略

本期采用“内部契约优先，集成回归兜底”的测试策略。

建议比例：

1. 单元测试约 45% 到 55%
2. 组件级 / service 级集成测试约 25% 到 35%
3. HTTP API 集成测试约 15% 到 25%
4. 前端契约与人工冒烟约 5% 到 10%

说明：

一期重点是接口闭环。
二期重点是平台边界。

因此本期测试必须比一期更偏内部结构和契约保护。

### 4.3 本期明确不做

本期不做以下高成本测试：

1. 真实底层邮箱来源的全自动端到端联调
2. 浏览器级完整 E2E 自动化
3. 压测级 provider 并发测试
4. 多 provider 同时在线切换的生产级演练

原因：

本期核心风险不是外部网络连不连得通，而是：

1. provider 抽象是否正确
2. mailbox model 是否正确
3. 上层契约是否真正与底层解耦

## 5. 测试分层设计

## 5.1 Provider Contract 单元测试

目标：验证底层 provider 适配层遵守统一内部契约，而不是把具体来源细节继续暴露给 service。

必须覆盖：

1. provider `get_options()` 返回统一 options 结构
2. provider `create_mailbox()` 返回统一 mailbox create 结果
3. provider 允许返回 provider-specific handle
4. provider `list_messages()` 返回统一 message summary 结构
5. provider `get_message_detail()` 返回统一 detail 结构
6. provider `delete_message()` / `clear_messages()` 返回稳定成功失败语义
7. provider 错误映射不直接泄漏原始上游异常文本

测试重点：

1. 不是测某个具体 provider 的网络行为
2. 而是测“所有 provider 都必须满足的内部 contract”

建议测试文件：

1. `tests/test_temp_mail_provider_contract.py`

## 5.2 Mailbox Internal Model 测试

目标：验证 temp mailbox 在本项目中已经是正式资源对象，而不只是 `email_addr` 字符串。

必须覆盖：

1. `temp_emails` 记录能保存并读取 `meta_json`
2. `meta_json` 可承载 provider-specific handle
3. mailbox repo 返回值中保留完整 mailbox 记录，而不是被简化成裸邮箱
4. 用户邮箱与任务邮箱共享同一 mailbox model，只在业务字段上有差异
5. `visible_in_ui / mailbox_type / task_token / consumer_key / status / meta_json` 可同时存在并正确读写

建议测试文件：

1. `tests/test_temp_mail_mailbox_model.py`

## 5.3 TempMailService 单元与组件测试

目标：验证 service 已真正成为“业务语义层”，而不是继续和单一 provider 强绑定。

必须覆盖：

1. `TempMailService` 通过 provider factory 获取 provider，而不是默认硬绑定单类
2. service 创建用户邮箱时，会保存 mailbox record 而非仅保存 email
3. service 创建任务邮箱时，会保存任务字段和 provider handle
4. service 读取消息时，先加载 mailbox record，再调用 provider
5. service 删除邮箱时，先按 mailbox record 调 provider，再做本地删除或同步清理
6. service 提取验证码仍使用统一 extractor，不感知底层 provider
7. finish 后 task mailbox 状态与 external 读取拒绝语义不变

建议测试文件：

1. `tests/test_temp_mail_service_platform.py`

## 5.4 Provider Factory 测试

目标：验证 runtime 确实已经具备 provider 选择能力，而不是继续停留在单实现默认绑定。

必须覆盖：

1. settings 指向某 provider 时，factory 返回对应实现
2. 未配置 provider 时，返回默认实现或明确错误
3. 非法 provider 值返回稳定错误
4. legacy bridge 被视为兼容 provider，而不是正式产品语义

建议测试文件：

1. `tests/test_temp_mail_provider_factory.py`

## 5.5 Settings 契约与 Runtime 一致性测试

目标：验证 settings、env fallback、runtime 选择语义是一套语言，而不是表面 temp-mail、底层 legacy 混杂。

必须覆盖：

1. `/api/settings` 继续只暴露正式 `temp_mail_*` 契约
2. provider-specific 配置可以被保存并读取
3. legacy 兼容配置不会反向污染正式字段
4. runtime 读取 settings 后，provider factory 能得到一致结果
5. legacy env fallback 只作为兼容，不应覆盖显式正式配置

建议测试文件：

1. `tests/test_temp_mail_settings_platform_contract.py`

## 5.6 External 兼容回归测试

目标：验证即使底层 provider 平台化，上层 external 读取契约依然保持不变。

必须覆盖：

1. `apply -> messages -> verification -> finish` 继续成立
2. `messages/latest`、`messages/{id}`、`verification-code`、`verification-link` 继续成立
3. task mailbox 的 `consumer_key` 授权继续成立
4. finish 后 `TASK_FINISHED` / `PROBE_CANCELLED` 等稳定终态继续成立
5. 普通 Outlook / IMAP external 路径不被误伤

说明：

这层不是新功能测试，而是二期平台化之后最重要的“上层不变”证明。

建议沿用并补强：

1. `tests/test_external_api_temp_mail_compat.py`

## 6. 测试选型说明

## 6.1 为什么本期要新增 Provider Contract 测试

因为二期目标不是“再写一个 provider 文件”，而是“让 provider 成为内部正式能力边界”。

如果没有 contract test，就只能证明某个实现暂时能用，不能证明整个平台的边界已经稳定。

## 6.2 为什么本期要新增 Mailbox Model 测试

因为本期最关键的设计变化，就是 mailbox 不再只是 `email_addr`。

如果没有 mailbox model 测试，那么：

1. `meta_json` 很容易再次沦为“预留字段”
2. service 也很容易继续偷懒按裸邮箱地址驱动

## 6.3 为什么本期 external 测试仍然必须保留高优先级

因为 temp-mail 的平台化如果只证明内部抽象更漂亮，但 external 被打爆，那这个重构在产品上就是失败的。

external 兼容仍然是最重要的上层不回归证明。
