## 问题描述

当前数据库中，以下关键字段以**明文**存储：

| 字段 | 位置 | 风险 |
|---|---|---|
| `password`（IMAP 密码） | `accounts` 表 | ❌ 数据库泄露 → 账号直接可被访问 |
| `refresh_token`（Microsoft Refresh Token） | `accounts` 表 | ❌ 数据库泄露 → 账号 Token 可直接使用 |
| `cf_worker_admin_key`（CF Worker 管理密码） | `settings` 表 | ❌ 可直接操控 CF Worker |

对比：以下字段已加密，做得较好：
- `login_password`：bcrypt 哈希 ✅
- `telegram_bot_token`：`enc:` AES-256 加密 ✅
- `external_api_key`：`enc:` AES-256 加密 ✅

## 推荐方案：应用层 AES-256 加密（`enc:` 前缀方案）

### 现有基础设施

`outlook_web/security/crypto.py` 中已有：
- `encrypt_data(value: str) -> str`：返回 `enc:<base64>` 格式
- `decrypt_data(value: str) -> str`：自动识别 `enc:` 前缀并解密
- `is_encrypted(value: str) -> bool`：判断是否已加密
- 加密密钥来自 `SECRET_KEY` 环境变量

### 实施步骤

#### 步骤 1：写入时加密（accounts 控制器）

在 `controllers/accounts.py` 的账号创建/更新逻辑中：
```python
# 修改前
account_data["password"] = raw_password
account_data["refresh_token"] = raw_token

# 修改后
from outlook_web.security.crypto import encrypt_data, is_encrypted
account_data["password"] = encrypt_data(raw_password) if raw_password and not is_encrypted(raw_password) else raw_password
account_data["refresh_token"] = encrypt_data(raw_token) if raw_token and not is_encrypted(raw_token) else raw_token
```

#### 步骤 2：读取时解密（services 层）

在 `graph.py`、`imap.py`、`imap_generic.py` 读取凭据前：
```python
from outlook_web.security.crypto import decrypt_data, is_encrypted

def _resolve_credential(value: str) -> str:
    if not value:
        return ""
    return decrypt_data(value) if is_encrypted(value) else value
```

#### 步骤 3：刷新后更新加密写回（refresh.py）

Token 刷新后写回数据库时，加密后再存：
```python
new_encrypted_token = encrypt_data(new_refresh_token)
repo.update_account_token(account_id, new_encrypted_token)
```

#### 步骤 4：settings 中 CF Worker Admin Key 加密

与 external_api_key 保持一致：
```python
# 写入时
encrypted_key = encrypt_data(new_cf_key)
settings_repo.set_setting("cf_worker_admin_key", encrypted_key)

# 读取时
raw = get_setting("cf_worker_admin_key", "")
return decrypt_data(raw) if is_encrypted(raw) else raw
```

#### 步骤 5：历史数据迁移

需要一个一次性迁移脚本，把已有的明文凭据加密后更新：
```python
# 伪代码
for account in all_accounts:
    if account.password and not is_encrypted(account.password):
        db.execute("UPDATE accounts SET password = ? WHERE id = ?",
                   (encrypt_data(account.password), account.id))
    if account.refresh_token and not is_encrypted(account.refresh_token):
        db.execute("UPDATE accounts SET refresh_token = ? WHERE id = ?",
                   (encrypt_data(account.refresh_token), account.id))
```

可以作为数据库 schema 迁移（版本 18）的一部分自动执行。

## 注意事项

1. **SECRET_KEY 必须持久化**：`SECRET_KEY` 一旦改变，所有已加密的凭据都无法解密。必须在部署时固定 SECRET_KEY 并妥善保存。
2. **向后兼容**：读取时检查 `is_encrypted()` 实现软迁移——旧明文数据仍然可读，新数据加密写入。
3. **导出时不能包含明文**：账号导出功能（`/api/export`）应移除 password/refresh_token 字段，或加密后导出。
4. **回归测试**：改动影响账号的读写全链路（导入、刷新、读邮件），需要完整的集成测试覆盖。

## 暂不处理的问题（已记录讨论结论）

### 高并发 / 上万账号 Token 刷新

- 当前顺序刷新，10,000 账号约需 19 小时
- 改进方向：线程池并发刷新 + 分批调度
- **结论：当前规模不需要，暂不处理**

### SQLite 切换 PostgreSQL

- SQLite 单写者模型在高并发写时有瓶颈
- 改进方向：切换 PostgreSQL 或开启 WAL 模式
- **结论：当前规模不需要，暂不处理**
