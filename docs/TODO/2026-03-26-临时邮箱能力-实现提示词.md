## 3. 当前项目真实定位

当前项目不是空白状态，也不是一期未完成状态。

真实现状是：

1. 一期 temp-mail 产品能力已经基本可用。

当前核心问题不再是：

1. 页面有没有入口

当前核心问题是：

1. `TempMailProviderBase` 仍偏向 `email_addr` 参数契约
2. `TempMailService` 仍默认绑定单 provider
3. `meta_json` 还不是正式 mailbox handle 契约
4. runtime provider 选择层还不清晰
5. external / service / settings 还没有完全围绕统一 mailbox descriptor 收口

## 4. 你必须遵守的关键实现约束

### 4.1 不要做错方向

以下几条是二期硬约束：

1. 不要重新设计一套新的用户侧 temp-mail API。
2. 不要新增 `/api/external/temp-emails/{task_token}/messages*` 这一整套重复路径。

5. 不要只做“名字改了、文案改了”的表面收口，而要完成内部边界收口。

### 4.2 二期要优先改内部接口，不是外部路径

二期最关键的是：

1. mailbox descriptor
2. provider base contract
3. provider factory
4. mailbox-first 的 `TempMailService`
5. resolver 输出统一 descriptor
6. settings/runtime 选择语义

### 4.3 上层产品契约必须稳定

以下接口必须保持可用且不改路径：

1. `/api/temp-emails/*`
2. `POST /api/external/temp-emails/apply`
3. `POST /api/external/temp-emails/{task_token}/finish`
4. `/api/external/messages*`
5. `/api/external/verification-*`
6. `/api/external/wait-message`
7. `/api/settings`

### 4.4 用户和任务侧继续共享统一语义

必须继续保持：

1. 用户邮箱可见、任务邮箱不可见
2. 用户侧和 external 侧复用同一套 verification 结果语义
