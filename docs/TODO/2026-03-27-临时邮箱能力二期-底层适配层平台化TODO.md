## 1. 文档目标

本文档用于把二期 PRD 中“底层适配层平台化”目标拆成可执行任务。

二期不是再补一轮 temp-mail 页面功能，而是完成以下技术收口：

1. mailbox 从邮箱字符串升级为正式内部资源。
2. provider 从单实现桥接升级为统一 contract + factory。
3. service 从 provider 直通层升级为 mailbox-first 业务语义层。
4. external 读取从“邮箱分类判断”升级为“descriptor 驱动分流”。
5. settings / env / runtime 形成一致语义。

## 2. 当前结论

### 2.1 一期已完成，作为二期前提

1. `/api/temp-emails/*` 用户侧产品能力已存在。
2. `apply / finish` 已存在。
3. `/api/external/messages*` 已能兼容 temp mailbox。
4. `temp_mail_*` 已成为正式 settings 语义。
5. `GPTMail` 已不再作为正式产品能力对外暴露。

### 2.2 当前剩余观察项

当前二期主链已收口完成，剩余项以下一阶段硬化为主：

1. 少量历史用户侧响应仍沿用固定 provider 展示值；后续若接入第二正式 provider，建议继续统一到公共 dto 输出。
2. `legacy_bridge` 当前通过 factory + meta 隔离，但底层仍复用 `CustomTempMailProvider` 实现；后续若引入真正独立 legacy bridge，可继续拆分。
3. 人工环境验收仍建议在真实部署配置下执行一次，以补足自动化覆盖之外的运行时验证。

### 2.3 当前执行原则

1. 不改正式产品路径。
2. 不新增第二套 external temp-mail 读信协议。
3. 优先改内部边界，再看页面层。
4. 优先补 contract tests，再做重构。

### 2.4 当前实施状态

当前状态已从“二期待开发”进入“二期主线完成并已回归验证”：

1. mailbox descriptor、repo 完整 record、`meta_json` schema 与 public dto 已落地。
2. provider base、provider factory、custom provider、legacy bridge 边界与错误语义已收口。
3. `TempMailService`、`mailbox_resolver`、`external_api`、settings/runtime 一致性改造已完成。
4. 专项测试、兼容回归、全量 `discover` 已全部跑通。
5. 本 TODO 当前主要用于记录完成状态与遗留观察项，不再作为开发启动入口。

## 2.5 最终交付物定义

二期最终要交付的不是“更多 temp-mail 接口”，而是下面这套完整结果：

1. 对用户和注册机来说，上层 temp-mail 产品能力保持不变。
2. 对系统内部来说，mailbox、provider、service、resolver、settings 五层边界已经稳定。
3. provider 已可以在 runtime 被选择和替换，而不会反向污染上层 API。
4. legacy bridge 即使继续保留，也只存在于内部兼容层。
5. 测试能够证明“上层稳定、底层可替换”已经成立。
