# FD: 临时邮箱能力

- 文档版本: v0.1
- 创建日期: 2026-03-26
- 关联 PRD: `docs/PRD/2026-03-26-临时邮箱能力-功能PRD.md`
- 当前范围: 仅讨论前端表现、后端接口行为与功能集成

## 1. 功能定义

本功能用于将项目现有临时邮箱能力从 `GPTMail` 来源切换为“自建临时邮箱项目”来源，并完成前端页面、后端统一接口与底层邮箱数据之间的集成。

对于用户和注册机来说，项目仍然只提供一套统一的临时邮箱能力，包括：

1. 生成邮箱
2. 查看邮件
3. 查看邮件详情
4. 提取验证码
5. 提取验证链接
6. 删除邮件
7. 清空邮箱
8. 删除邮箱

## 2. 功能范围

### 2.1 本期包含

1. 保留现有“临时邮箱”页面入口
2. 替换临时邮箱底层数据来源
3. 保持用户侧基本操作习惯不变
4. 为注册机提供统一接口能力
5. 支持自定义前缀
6. 支持多域名策略 B

### 2.2 本期不重点展开

1. 复杂邮箱生命周期设计
2. 复杂任务状态机
3. 自动回收策略细节
4. Cloudflare 自动部署与自动配置流程

## 3. 角色与可见范围

### 3.1 普通用户

普通用户只能看到并操作自己手动创建的临时邮箱。

普通用户可执行的操作：

1. 创建临时邮箱
2. 查看临时邮箱列表
3. 查看邮件列表
4. 查看邮件详情
5. 提取验证码
6. 提取验证链接
7. 删除邮件
8. 清空邮箱
9. 删除邮箱
10. 复制邮箱地址

### 3.2 注册机 / 自动化调用方

注册机通过统一接口申请邮箱并读取结果。

注册机创建的邮箱：

1. 默认不展示在前端页面
2. 只通过接口返回给调用方
3. 在能力上与普通临时邮箱一致

### 3.3 管理员

管理员负责启用和配置临时邮箱能力，但本 FD 不展开配置后台的详细页面设计。

## 4. 前端页面定义

### 4.1 页面入口

继续使用当前项目中的“临时邮箱”页面入口。

前端不新增第二套独立产品入口，不要求用户理解底层来源变化。

### 4.2 页面结构

临时邮箱页面保持三块核心区域：

1. 邮箱列表区
2. 邮件列表区
3. 邮件详情区

### 4.3 邮箱列表区

邮箱列表区用于展示用户手动创建的临时邮箱。

每个邮箱卡片至少展示：

1. 邮箱地址
2. 临时邮箱标识
3. 提取验证码入口
4. 复制入口
5. 清空入口
6. 删除入口

页面要求：

1. 不展示接口创建的任务邮箱
2. 支持选中当前邮箱
3. 支持空态展示

### 4.4 邮件列表区

邮件列表区用于展示当前选中邮箱下的邮件。

每封邮件至少展示：

1. 发件人
2. 主题
3. 摘要
4. 时间

页面要求：

1. 进入邮箱后可主动获取或刷新邮件
2. 邮件为空时有明确空态提示
3. 加载失败时有明确错误提示

### 4.5 邮件详情区

邮件详情区用于展示单封邮件的完整内容。

详情展示至少包括：

1. 发件人
2. 收件邮箱
3. 主题
4. 时间
5. 正文内容
6. HTML 内容或文本内容

页面要求：

1. 支持展示可读正文
2. 支持验证码/链接提取结果在详情区或工具区展示
3. 加载失败时有明确错误反馈

## 5. 用户侧交互流程

### 5.1 创建邮箱流程

用户进入临时邮箱页面后，可创建一个新的临时邮箱。

创建时的前端输入规则：

1. 用户可不填前缀
2. 用户可填写自定义前缀
3. 如果系统存在多个域名，前端默认选择“自动分配”
4. 用户也可手动选择指定域名

创建成功后：

1. 新邮箱加入当前前端邮箱列表
2. 页面提示创建成功
3. 用户可立即点击进入该邮箱查看邮件

### 5.2 查看邮件流程

1. 用户选择某个邮箱
2. 系统加载该邮箱的邮件列表
3. 用户点击某封邮件
4. 系统展示邮件详情

### 5.3 提取验证码流程

1. 用户选择邮箱或邮件
2. 用户点击“提取验证码”
3. 系统返回验证码结果
4. 前端将结果展示给用户

前端集成要求补充：

1. 临时邮箱页面不能继续复用普通邮箱的提取请求路径
2. 如果前端当前使用共享函数处理“复制验证码”，该共享函数必须支持按邮箱类型分流
3. 临时邮箱分支必须请求 `/api/temp-emails/<email_addr>/extract-verification`
4. 普通邮箱分支才继续请求 `/api/emails/<email_addr>/extract-verification`

### 5.4 提取验证链接流程

1. 用户选择邮箱或邮件
2. 用户点击“提取验证链接”
3. 系统返回链接结果
4. 前端将结果展示给用户

### 5.5 删除与清空流程

1. 用户可删除单封邮件
2. 用户可清空当前邮箱全部邮件
3. 用户可删除当前邮箱

前端要求：

1. 删除与清空操作需有确认提示
2. 操作成功后页面列表应立即刷新
3. 当前邮箱被删除后，页面应回到未选中状态

## 6. 注册机侧接口流程

### 6.1 申请邮箱

注册机调用统一接口申请邮箱。

接口调用规则：

1. 可不传前缀
2. 可传指定前缀
3. 可不传域名
4. 可传指定域名
5. 不传域名时默认自动分配

返回结果至少包括：

1. 邮箱地址
2. 当前任务使用所需的标识信息
3. 请求结果状态

### 6.2 查询邮件

注册机不单独新增一套任务邮箱专用读信接口。

注册机在申请到任务邮箱后，继续复用项目现有对外开放读信接口完成：

1. 查询邮件列表
2. 查询最新邮件
3. 获取验证码
4. 获取验证链接

返回结果应与当前外部接口保持一致，避免形成第二套长期并存的读信协议。

### 6.3 获取验证码 / 链接

注册机在拿到任务邮箱地址后，可继续通过现有对外开放接口：

1. 获取验证码
2. 获取验证链接

接口返回规则：

1. 返回成功时给出明确结果
2. 未匹配到邮件或未匹配到内容时给出明确失败信息

### 6.4 结束使用

注册机在任务完成后可调用结束接口。

本期只要求提供“结束当前任务使用”的接口能力，不要求在 FD 中定义复杂状态流转。

## 7. 前后端接口定义

### 7.1 用户侧接口

继续使用当前临时邮箱接口路径，统一由本项目后端提供。

当前接口集合：

1. `GET /api/temp-emails`
   - 获取用户可见邮箱列表
2. `GET /api/temp-emails/options`
   - 获取临时邮箱创建配置项
3. `POST /api/temp-emails/generate`
   - 创建临时邮箱
4. `GET /api/temp-emails/<email_addr>/extract-verification`
   - 提取验证码/验证链接
5. `DELETE /api/temp-emails/<email_addr>`
   - 删除临时邮箱
6. `GET /api/temp-emails/<email_addr>/messages`
   - 获取邮件列表
7. `GET /api/temp-emails/<email_addr>/messages/<message_id>`
   - 获取邮件详情
8. `DELETE /api/temp-emails/<email_addr>/messages/<message_id>`
   - 删除单封邮件
9. `DELETE /api/temp-emails/<email_addr>/clear`
   - 清空邮箱
10. `POST /api/temp-emails/<email_addr>/refresh`
   - 刷新邮件

### 7.2 用户侧接口行为调整

为满足新需求，用户侧接口应满足以下行为：

1. 列表接口只返回用户手动创建的邮箱
2. `options` 接口返回域名列表、默认策略和前缀规则
3. 创建接口支持接收 `prefix`
4. 创建接口支持接收 `domain`
5. 当存在多个域名时，`domain` 可为空，表示自动分配
6. 提取接口统一返回验证码与验证链接结果
7. 邮件相关接口继续按用户当前选中的邮箱返回结果
8. 前端共享提取函数如果同时服务普通邮箱和临时邮箱，必须具备 source-aware 分流能力

### 7.3 注册机侧接口

注册机侧不复用前端列表展示逻辑，但也不重复发明一整套新的读信接口。

注册机侧最小新增接口只需要支持：

1. 申请邮箱
2. 结束使用

注册机在申请到任务邮箱之后，读邮件与提取能力继续复用现有外部接口：

1. `GET /api/external/messages`
2. `GET /api/external/messages/latest`
3. `GET /api/external/verification-code`
4. `GET /api/external/verification-link`

注册机接口行为规则：

1. 接口创建的任务邮箱不写入前端展示列表
2. 申请接口返回的数据字段应稳定且适合自动化调用
3. 支持传入前缀和域名
4. 任务邮箱申请成功后，应能被现有 external 读信接口直接读取
5. `finish` 后，对该任务邮箱的普通 external 读取请求应返回拒绝
6. 若任务邮箱已存在进行中的异步等待或 probe，请在技术实现中定义统一的结束后行为

## 8. 数据展示与转换规则

### 8.1 后端职责

后端负责把底层自建临时邮箱项目返回的数据，转换为当前项目前端和注册机可直接消费的统一结构。

### 8.2 用户侧邮件列表统一字段

邮件列表至少统一为以下字段：

1. `id`
2. `from`
3. `subject`
4. `body_preview`
5. `date`
6. `timestamp`
7. `has_html`

### 8.3 用户侧邮件详情统一字段

邮件详情至少统一为以下字段：

1. `id`
2. `from`
3. `to`
4. `subject`
5. `body`
6. `body_type`
7. `date`
8. `timestamp`

### 8.4 接口侧结果统一原则

1. 用户侧和注册机侧的验证码/链接提取能力保持统一
2. 前端关注可展示字段
3. 注册机关注稳定结果字段
4. 底层来源变化不应直接暴露给前端用户

## 9. 状态与反馈规则

### 9.1 加载态

以下场景需要加载态：

1. 加载邮箱列表
2. 创建邮箱
3. 获取邮件列表
4. 获取邮件详情
5. 执行提取操作

### 9.2 空态

以下场景需要空态：

1. 没有任何临时邮箱
2. 当前邮箱没有邮件
3. 当前未选中任何邮件

### 9.3 成功反馈

以下操作需要成功反馈：

1. 创建邮箱成功
2. 删除邮件成功
3. 清空邮箱成功
4. 删除邮箱成功
5. 提取验证码成功
6. 提取链接成功

### 9.4 失败反馈

以下场景需要明确错误提示：

1. 创建邮箱失败
2. 指定域名不可用
3. 指定前缀不可用
4. 邮件拉取失败
5. 邮件详情加载失败
6. 未提取到验证码
7. 未提取到验证链接

## 10. 多域名与前缀行为定义

### 10.1 多域名策略

采用方案 B：

1. 默认自动分配域名
2. 同时允许手动选择域名

### 10.2 前端表现

当前端检测到有多个可用域名时：

1. 创建邮箱区域应展示域名选择项
2. 默认值为“自动分配”
3. 用户可切换为指定域名

当只有一个可用域名时：

1. 前端可不强调域名选择
2. 系统直接使用默认域名

### 10.3 前缀表现

1. 创建邮箱区域应提供可选前缀输入框
2. 用户不填写前缀时，系统自动生成
3. 用户填写前缀时，系统按输入尝试生成邮箱

## 11. 验收标准

### 11.1 用户侧

1. 用户可通过当前临时邮箱页面创建自建域名临时邮箱
2. 用户可看到自己手动创建的邮箱列表
3. 用户可查看邮件列表和邮件详情
4. 用户可提取验证码和验证链接
5. 用户可删除邮件、清空邮箱、删除邮箱
6. 用户可在多域名场景下默认自动分配域名，也可手动选择域名
7. 用户可在需要时输入自定义前缀

### 11.2 注册机侧

1. 注册机可通过统一接口申请邮箱
2. 注册机可通过统一接口获取验证码
3. 注册机可通过统一接口获取验证链接
4. 注册机可通过统一接口结束使用
5. 注册机创建的邮箱默认不出现在前端用户列表
6. 注册机可按需要传入指定前缀和指定域名
