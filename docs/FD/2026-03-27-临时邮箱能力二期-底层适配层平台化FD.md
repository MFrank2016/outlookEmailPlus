## 0. 当前实施状态

本期 FD 定义的用户可见行为已与实现对齐：

1. 用户侧 temp-mail 产品入口、任务侧 apply/finish、external messages/verification/wait-message 路径保持原样。
2. 用户邮箱与任务邮箱继续共用统一读取与提取语义，finish 后拒绝语义保持稳定。
3. 页面与 controller 仍只暴露 temp-mail 产品语义，不泄漏 provider-specific handle、cursor 或鉴权字段。
4. settings 继续以 `temp_mail_*` 为正式可见字段，runtime provider 已转入内部选择语义。
5. 回归已覆盖用户侧 API、external temp-mail 兼容矩阵和普通 Outlook/IMAP external 读取能力。

当前剩余关注点是后续若引入第二个正式 provider，需要继续统一少量历史响应中的 provider 展示字段，但这不影响本期 FD 收口。

## 1. 功能定义

本期功能不是再新增一个 temp-mail 页面能力，而是把现有 temp-mail 能力升级成项目内稳定的底层适配平台。

对用户和注册机来说，系统仍然只提供一套统一的临时邮箱产品能力：

1. 创建临时邮箱
2. 查看临时邮箱列表
3. 查看邮件列表与详情
4. 提取验证码与验证链接
5. 删除邮件、清空邮箱、删除邮箱
6. 申请任务邮箱并通过统一 external 接口读取

但对系统内部来说，本期需要补齐的是：

1. temp-mail 的底层来源接入方式
2. temp mailbox 的正式内部资源模型
3. 上层产品契约与底层来源之间的隔离边界

本期完成后，temp-mail 在项目内应表现为：

1. 上层产品能力稳定
2. 底层来源可替换
3. 用户和任务侧不感知底层来源变化

## 2. 功能范围

### 2.1 本期包含

1. 维持现有 `/api/temp-emails/*` 用户侧正式路径
2. 维持现有 `/api/external/*` 任务侧正式路径
3. 维持用户邮箱可见、任务邮箱隐藏的现有产品规则
4. 将底层临时邮箱来源收口到统一内部适配层
5. 将 temp mailbox 升级为正式内部资源对象
6. 让 settings 与 runtime 语义保持一致

### 2.2 本期不重点展开

1. 新的 temp-mail 页面视觉改版
2. 独立的 provider 管理后台
3. 多 provider 可视化切换界面
4. 自动化域名开通或 DNS 编排
5. 第三方 provider 平台市场化能力

## 3. 角色与可见范围

### 3.1 普通用户

普通用户继续只感知 temp-mail 产品能力，不感知底层 provider 变化。

普通用户可见能力保持为：

1. 创建临时邮箱
2. 查看用户自己的临时邮箱列表
3. 查看邮件列表
4. 查看邮件详情
5. 提取验证码
6. 提取验证链接
7. 删除邮件
8. 清空邮箱
9. 删除邮箱
10. 复制邮箱地址

普通用户不可见：

1. 底层来源的鉴权字段
2. provider-specific 状态
3. 任务邮箱
4. 外部来源的原始接口路径

### 3.2 注册机 / 自动化调用方

注册机继续通过统一接口申请和使用任务邮箱，不感知底层来源变化。

注册机可执行：

1. 申请任务邮箱
2. 查询邮件列表
3. 查询最新邮件
4. 获取验证码
5. 获取验证链接
6. 结束任务邮箱使用

注册机不可见：

1. 底层 provider 的直接读信接口
2. temp mailbox 的前端页面模型
3. provider-specific 内部句柄

### 3.3 管理员

管理员继续通过项目 settings 配置 temp-mail 能力。

本期管理员关注的是：

1. temp-mail 是否启用
2. temp-mail 使用哪个正式 provider
3. temp-mail 域名、默认域名、前缀规则配置是否正确
4. provider-specific 配置是否已保存

管理员不应直接面对：

1. 底层 provider 的原始协议细节
2. 临时调试字段
3. 兼容期 legacy 产品语义

## 4. 系统功能表现

## 4.1 上层产品契约稳定

本期必须保持以下正式能力和入口不变：

### 用户侧

1. `/api/temp-emails`
2. `/api/temp-emails/options`
3. `/api/temp-emails/generate`
4. `/api/temp-emails/{email_addr}/messages`
5. `/api/temp-emails/{email_addr}/messages/{message_id}`
6. `/api/temp-emails/{email_addr}/extract-verification`
7. 删除 / 清空 / 刷新等路径

### 任务侧

1. `POST /api/external/temp-emails/apply`
2. `POST /api/external/temp-emails/{task_token}/finish`

### 读取侧

1. `GET /api/external/messages`
2. `GET /api/external/messages/latest`
3. `GET /api/external/messages/{message_id}`
4. `GET /api/external/verification-code`
5. `GET /api/external/verification-link`
6. `GET /api/external/wait-message`

功能要求：

1. 上述路径不因底层 provider 变化而发生路径层重构
2. 用户和注册机不需要学习第二套协议

## 4.2 temp mailbox 资源模型正式化

本期系统中的 temp mailbox 不再只表现为一个邮箱字符串。

在功能层面，系统必须支持把一个 temp mailbox 视为正式资源对象，至少包含：

1. 邮箱地址
2. 用户邮箱或任务邮箱类型
3. 可见性
4. 状态
5. 前缀
6. 域名
7. 任务标识信息
8. provider 内部句柄信息

对产品行为的影响：

1. 用户侧和任务侧都继续使用统一 mailbox 语义
2. 底层来源所需的附加信息属于内部能力，不对用户可见
3. service 与 external 读取逻辑不再只按邮箱字符串理解 temp mailbox

## 4.3 底层 provider 隔离

功能表现要求如下：

1. controller 层只调用 temp-mail service
2. 前端不直接调用底层来源接口
3. external 读取层不直接依赖底层来源协议
4. settings 页面不直接暴露底层来源原始字段名

从产品功能表现看，这意味着：

1. temp-mail 的正式行为统一由主项目输出
2. 底层来源变化不应改变用户操作路径
3. 底层来源变化不应改变注册机使用路径

## 4.4 settings 行为表现

用户或管理员在项目内看到的正式配置语义，仍应统一为 `temp_mail_*`。

页面和接口行为要求：

1. 设置页继续读取 temp-mail 正式字段
2. 设置页继续保存 temp-mail 正式字段
3. provider-specific 配置可被保存，但不应破坏正式 temp-mail 语义
4. 兼容期 legacy 配置只作为内部兼容，不作为正式产品展示

用户可理解的配置结果包括：

1. 当前启用哪个 temp-mail provider
2. 当前允许哪些域名
3. 当前默认域名是什么
4. 前缀规则是什么

## 5. 用户侧功能表现

## 5.1 创建邮箱流程

用户进入临时邮箱页面后，可继续创建新的临时邮箱。

前端交互结果应保持：

1. 用户可不填前缀
2. 用户可填写前缀
3. 用户可选择“自动分配域名”
4. 多域名时用户可手动选择目标域名

创建成功后：

1. 新邮箱进入用户列表
2. 页面提示创建成功
3. 用户可立即进入该邮箱查看邮件

本期新增的功能要求不是前端新增控件，而是：

1. 创建动作必须继续只通过主项目 temp-mail 接口完成
2. 不允许前端因为接入底层 provider 而直接发起 provider 请求

## 5.2 查看邮件与详情

用户侧仍然表现为统一的邮箱工作台：

1. 选择邮箱
2. 查看邮件列表
3. 打开邮件详情
4. 在详情区提取验证码或链接

本期对功能的要求是：

1. 即使底层 provider 更换，用户侧查看结果语义不变
2. 详情展示仍按项目自己的正文/HTML/附件处理逻辑输出
3. temp-mail 页面不应暴露 provider-specific 原始数据结构

## 5.3 提取能力

提取功能继续保持统一语义。

对用户表现要求：

1. 提取验证码入口仍可用
2. 提取验证链接入口仍可用
3. 返回结果字段含义不变化
4. temp mailbox 的提取结果与任务邮箱读取的提取结果保持一致

## 5.4 删除与清理

用户侧继续具备：

1. 删除单封邮件
2. 清空邮箱
3. 删除邮箱

本期新增要求：

1. 这些动作的上层行为继续不变
2. 即使底层 provider 有自己的句柄或删除方式，主项目仍然输出统一结果语义

## 6. 任务侧功能表现

## 6.1 apply

注册机继续只通过：

1. `POST /api/external/temp-emails/apply`

申请任务邮箱。

返回结果至少保持：

1. `email`
2. `task_token`
3. `visible_in_ui=false`

本期新增要求：

1. task mailbox 在内部也必须具备完整 mailbox model
2. provider-specific handle 只属于内部实现，不暴露给 external 调用方

## 6.2 读取

任务邮箱继续复用现有 external 读取体系：

1. 查询列表
2. 查询最新邮件
3. 查询详情
4. 提取验证码
5. 提取链接
6. 等待邮件

功能表现要求：

1. 不新增 temp-mail 专属第二套读信协议
2. 不因底层 provider 替换而改 external 路径
3. `consumer_key` 归属、finish 后拒绝、probe/wait 行为继续成立

## 6.3 finish

任务邮箱继续只通过：

1. `POST /api/external/temp-emails/{task_token}/finish`

结束。

功能表现要求：

1. finish 继续只改变主项目内的任务邮箱语义状态
2. finish 后 external 新读取请求继续返回稳定拒绝
3. finish 后等待与 probe 的终态语义保持稳定

## 7. 页面与系统交互要求

## 7.1 前端对底层来源无感

本期页面层必须满足：

1. 临时邮箱页面继续只感知 temp-mail 正式接口
2. 正常邮箱页面与 temp-email 页面继续只按产品语义区分，不按底层 provider 区分
3. 页面代码中不应引入新的底层 provider URL、header 或品牌语义

## 7.2 用户可见内容不泄漏底层实现

本期必须避免以下用户可见行为：

1. 页面直接出现底层 provider 原始品牌
2. 页面直接展示 provider-specific token / id / auth 信息
3. 页面因更换底层来源而出现新的使用说明分支

## 8. 功能完成标准

本期功能在产品层面完成，至少意味着：

1. temp-mail 继续表现为一个稳定产品能力，而不是单一上游的包装层
2. 用户和注册机不需要学习新的路径或新的使用方式
3. settings 与 runtime 语义能够对齐
4. 用户邮箱、任务邮箱、external 读取和提取能力继续闭环
5. 底层来源变化不会直接扩散到 controller、前端和 external 契约

## 9. 当前确认结论

1. 一期 FD 解决的是“temp-mail 产品闭环表现”
2. 本期 FD 解决的是“temp-mail 平台化后，上层系统到底应该表现出什么功能结果”
3. 本期重点已经不再是页面上有没有某个按钮，而是整套系统是否真正具备“上层稳定、底层可替换”的产品行为
