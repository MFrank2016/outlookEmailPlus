# 账号管理简洁模式 TD

## 1. 文档目的

本文档用于冻结“账号管理简洁模式”当前已确认的交互模板，并给出基于现有前端结构的落地方案。

本文档不是继续发散方案，而是把已经确认的产品口径收敛成后续开发可直接执行的技术设计。

当前冻结模板来源：

- 交互预览：

## 2. 已确认的产品口径

### 2.1 模式定位

简洁模式不是替代当前标准三栏模式，而是账号管理页下的第二种工作模式。

目标是降低“取码、复制、打标签、拉取、删除”的操作阻力，适合高频连续点击场景。

### 2.2 与标准模式的关系

- 标准模式继续保留当前三栏结构：
  分组列 + 账号列 + 邮件列/详情
- 简洁模式改为单主工作台：
  分组选择条 + 账号列表工作台
- 两种模式共用同一套账号、分组、标签、批量操作数据源

### 2.3 简洁模式核心原则

- 不展示大段介绍说明
- 不展示搜索框
- 不默认依赖右侧邮件详情
- 每一行就是一个独立操作单元
- `拉取` 单独露出
- 其他低频操作折叠进三点菜单

### 2.4 标签口径

简洁模式不新增标签系统，必须复用正常模式现有标签体系和接口：

- `管理标签`
- `打标签`
- `去标签`

对应复用现有前端函数与后端接口，不新建独立标签弹窗或标签接口。

## 3. 页面结构冻结

### 3.1 顶部区

顶部沿用现有账号管理页面壳子：

- 页面标题：`账号管理`
- 副标题：`简洁模式`
- 右上模式切换：
  - `标准模式`
  - `简洁模式`
- 保留 `添加账号`

### 3.2 主工作台结构

简洁模式主区域固定为三层：

1. 工具条
2. 分组选择条
3. 账号列表区

底部保留一条批量操作栏。

### 3.3 工具条

工具条展示：

- 当前模式标识：`简洁取码模式`
- 当前分组和账号数量
- `管理标签`

说明：

- `管理标签` 直接复用正常模式 `showTagManagementModal()`
- 批量危险操作不放在顶部工具条，避免与底部批量栏重复

### 3.4 分组选择条

分组区改为横向分组卡片，而不是左侧独立分组列。

每个分组卡片展示：

- 分组名称
- 分组说明/提示语
- 账号数量

交互要求：

- 点击分组卡片切换当前分组
- 当前分组高亮
- 切换后刷新下方账号列表
- 顶部“当前分组 / 数量”同步更新

### 3.5 账号列表表头

表头固定为：

- 选择框
- 邮箱
- 验证码
- 最新邮件
- 标签
- 操作

### 3.6 账号行结构

每一行必须固定为以下信息：

#### 邮箱列

展示账号身份信息：

- 邮箱地址
- provider
- 当前分组
- 状态

交互：

- 点击邮箱地址直接复制邮箱

#### 验证码列

展示最近一封可提取验证码邮件的验证码。

交互：

- 点击验证码直接复制验证码
- 没有验证码时展示 `暂无`

规则冻结：

- 第二列显示“最近可提取验证码”，不是“最新一封邮件的正文截断”
- 取值范围为当前账号收件箱与垃圾邮件中，最近一封成功提取出验证码的邮件
- 判定顺序按邮件接收时间倒序
- 如果最新邮件没有验证码，但更早一封邮件有验证码，则第二列仍展示最近可提取验证码

#### 最新邮件列

展示最近邮件摘要：

- 主题
- 文件夹/时间/发件人

注意：

- 这里只展示摘要，不展开正文
- 简洁模式下不再默认打开详情栏
- 该列始终展示当前账号最新一封邮件摘要，无论该邮件是否包含验证码

#### 标签列

展示当前账号已有标签。

要求：

- 标签显示风格允许简化
- 标签数据必须仍来自现有 `acc.tags`

#### 操作列

操作列固定为：

- `拉取`
- 三点菜单

三点菜单固定为：

- `复制邮箱`
- `复制验证码`（有验证码时）
- `打标签`
- `移动分组`
- `删除账号`

## 4. 批量操作设计

简洁模式底部保留批量操作栏，触发条件为全局存在已选账号。

批量操作项固定为：

- `打标签`
- `去标签`
- `移动分组`
- `批量删除`

行为要求：

- 批量选择逻辑复用当前 `selectedAccountIds`
- `selectedAccountIds` 作为账号管理页级共享选择状态，标准模式和简洁模式共用
- 底部批量栏显示全局已选数量
- 当前分组的全选框只作用于当前可见分组账号
- 分组切换后，不清空其他分组已选项

- `showBatchDeleteConfirm()`

参考文件：

### 5.2 不应复用的 UI 结构

以下结构不应直接搬进简洁模式：

- 左侧独立分组列
- 账号列顶部搜索框
- 排序按钮条
- 右侧邮件列表/邮件详情联动区域
- 账号卡片大面积信息块

原因：

- 会破坏“低阻力、连续点击”的目标
- 会把简洁模式重新做回标准模式的变体

## 6. 前端接入方案

## 6.1 页面组织建议

在 `账号管理` 页面内部增加模式状态，而不是新建独立页面。

建议新增状态：

- `mailboxViewMode = 'standard' | 'compact'`

默认建议：

- 首次仍进入 `standard`
- 用户切换后本地持久化最近选择

### 6.2 模板层改造

改造文件：

- [index.html](/E:/hushaokang/Data-code/outlookEmail/templates/index.html)

建议方案：

1. 在 `page-mailbox` 顶部区域加入模式切换入口
2. 标准模式容器保留现有 `mailbox-layout`
3. 新增一个简洁模式容器，例如：
   - `mailbox-compact-layout`
4. 通过 class 或 `display: none` 切换两种容器

### 6.3 JS 侧状态与渲染职责

建议新增简洁模式专属渲染模块，例如：

- `static/js/features/mailbox_compact.js`

职责建议：

- `renderCompactGroupStrip(groups, currentGroupId)`
- `renderCompactAccountList(accounts)`
- `switchMailboxViewMode(mode)`
- `syncCompactSelectionState()`
- `bindCompactActions()`

这样做的原因：

- 避免把当前 `groups.js` 继续堆成一个超大文件
- 简洁模式可以独立演进
- 标准模式风险更低

### 6.4 数据流建议

短期内不新增后端接口，直接复用现有接口返回。

简洁模式数据流：

1. 页面进入账号管理
2. 调用 `loadGroups()`
3. 用户切换到简洁模式
4. 复用当前分组选中状态
5. 调用已有 `loadAccountsByGroup(currentGroupId)`
6. 将结果交给简洁模式渲染器

### 6.5 标签复用方式

简洁模式触发标签操作时：

- `管理标签` -> `showTagManagementModal()`
- 批量 `打标签/去标签` -> `showBatchTagModal('add' | 'remove')`

单行 `打标签` 当前系统没有现成单账号标签入口。

建议做一个“复用批量标签弹窗的单账号适配层”。

即：

- 继续复用现有标签接口与弹窗
- 但弹窗调用时支持传入显式 `accountIds`
- 单行 `打标签` 传入当前账号 id
- 不清空用户当前已有批量选择

推荐形式：

- `showBatchTagModal('add', { scopedAccountIds: [accountId] })`

这样做的目的：

- 功能仍然复用正常模式标签体系
- 但不会破坏当前批量选择状态
- 更符合“低阻力连续操作”的原始 PRD

### 6.6 取码与复制

简洁模式中：

- 邮箱点击复用 `copyEmail(...)`
- 验证码点击优先复用 `copyVerificationInfo(...)` 或抽出更轻量的复制逻辑

建议：

- 将“取最近验证码摘要”的显示逻辑与“复制验证码”的动作分离
- 简洁模式列表只负责展示摘要和复制
- 真正的邮件拉取仍复用现有账号/邮件接口

规则补充：

- 第二列复制的是“最近可提取验证码”
- 第三列展示的是“最新邮件摘要”
- 这两列允许来自不同邮件，但规则必须固定且前后端一致

## 8. 风险与注意事项

### 8.1 风险一：继续把逻辑堆进 `groups.js`

当前 `groups.js` 已经承担：

- 分组加载
- 分组切换
- 账号列表渲染
- 搜索/筛选/排序

如果直接继续把简洁模式全部塞进去，后续维护成本会快速上升。

建议：

- 新增简洁模式专属模块
- 保留 `groups.js` 负责数据获取与通用能力

### 8.2 风险二：单行标签动作无现成入口

当前系统只有：

- 标签管理
- 批量打标签
- 批量去标签

没有单账号独立标签弹窗，也不能为了单行打标签去破坏当前已选状态。

因此本轮技术设计明确：

- 单行 `打标签` 复用批量打标签弹窗，但必须通过 scoped account ids 适配，不得清空全局已选集合

### 8.3 风险三：简洁模式与标准模式状态不同步

需要同步的核心状态：

- 当前分组
- 当前已选账号
- 标签数据缓存
- 模式切换状态
- 最近邮件摘要缓存

建议：

- 不在两个模式中各自维护一份独立数据源
- 两边共用当前页面全局状态

---

## 9. 验收标准

满足以下条件即可认为简洁模式正式接入完成：

1. 在账号管理页可看到 `标准模式 / 简洁模式` 切换入口
2. 切换到简洁模式后，页面显示分组条 + 列表工作台
3. 点击分组后，下方账号列表正确切换
4. 点击邮箱可复制邮箱
5. 点击验证码可复制验证码
6. 单行 `拉取` 可触发真实拉取动作
7. `管理标签 / 打标签 / 去标签` 复用现有标签体系
8. 批量删除复用现有逻辑
9. 单行 `打标签` 不清空当前批量选择状态
10. 第二列和第三列的取值规则前后端一致
11. 简洁模式不依赖右侧邮件详情栏即可完成主要操作

## 10. 后续开发建议

下一步不建议再继续改预览稿，而是直接进入“前端正式接入”阶段。

建议按下面顺序继续：

1. 在账号管理正式页面加入模式切换状态
2. 抽出简洁模式独立渲染模块
3. 接入真实分组和账号数据
4. 先打通标签复用和批量删除
5. 再补单行拉取与三点菜单动作

本文档定稿后，后续实现应以本文档和预览稿为准，不再重新讨论页面大结构。
