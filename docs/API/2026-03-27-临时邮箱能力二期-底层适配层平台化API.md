## 0. 当前实施状态

当前实现已满足本文件定义的二期 API 收口方向：

1. 未新增第二套 external temp-mail 读取路径，任务邮箱继续复用 `/api/external/messages*`、`/api/external/verification-*`、`/api/external/wait-message`。
2. `POST /api/external/temp-emails/apply` 与 `POST /api/external/temp-emails/{task_token}/finish` 正式路径保持稳定，返回体已补齐二期所需状态字段。
3. `/api/settings` 已固化 `temp_mail_*` 正式字段，并补充 `temp_mail_provider_label` 以区分用户可见语义和 runtime provider name。
4. 内部 API 已落地 mailbox descriptor、mailbox-first provider contract、provider factory、mailbox-first service 和 descriptor-driven resolver。
5. 相关契约已由专项测试与全量回归锁定，详细命令与结果见 `verification.md`。

## 1. 文档定位

本文件不是重新设计一套新的 temp-mail 对外 API。

本文件要解决的是二期 FD 真正关心的四件事：

1. 哪些用户侧和 external 正式路径必须保持不变。
2. 哪些现有接口虽然不改路径，但必须补字段、补行为约束或补错误语义。
3. `/api/settings` 为了支撑平台化到底还需要承载哪些正式配置字段。
4. 主项目内部为了完成 provider 平台化，必须补哪些“内部接口契约”。

一句话原则：

对外产品 API 尽量不变，对内适配层接口必须重构。

## 2. 二期接口结论总览

## 2.1 不应该新增的新对外路径

二期明确不新增以下对外产品路径：

1. 不新增 `/api/external/temp-emails/{task_token}/messages`
2. 不新增 `/api/external/temp-emails/{task_token}/messages/latest`
3. 不新增 `/api/external/temp-emails/{task_token}/verification-code`
4. 不新增 `/api/external/temp-emails/{task_token}/verification-link`
5. 不新增 provider 专属 `/api/provider/*` 前端可见接口

原因：

1. FD 要求任务邮箱继续复用现有 `/api/external/messages*`
2. FD 要求用户和注册机不感知底层 provider 更换
3. 二期重点是内部平台化，不是继续扩公开路径

## 2.2 需要保留不变的正式路径

### 用户侧

1. `GET /api/temp-emails`
2. `GET /api/temp-emails/options`
3. `POST /api/temp-emails/generate`
4. `GET /api/temp-emails/{email_addr}/messages`
5. `GET /api/temp-emails/{email_addr}/messages/{message_id}`
6. `GET /api/temp-emails/{email_addr}/extract-verification`
7. `DELETE /api/temp-emails/{email_addr}/messages/{message_id}`
8. `DELETE /api/temp-emails/{email_addr}/clear`
9. `DELETE /api/temp-emails/{email_addr}`
10. `POST /api/temp-emails/{email_addr}/refresh`

### 任务侧

1. `POST /api/external/temp-emails/apply`
2. `POST /api/external/temp-emails/{task_token}/finish`

### 统一读取侧

1. `GET /api/external/messages`
2. `GET /api/external/messages/latest`
3. `GET /api/external/messages/{message_id}`
4. `GET /api/external/verification-code`
5. `GET /api/external/verification-link`
6. `GET /api/external/wait-message`
7. `GET /api/external/probe/{probe_id}`

## 2.3 真正需要修改的接口类型

二期需要修改的不是“路径数量”，而是下面四类接口契约：

1. 用户侧正式接口的字段与行为约束
2. external 正式接口的 mailbox 解析与生命周期语义
3. `/api/settings` 的正式配置字段
4. 主项目内部的 provider / service / mailbox descriptor 接口

## 3. FD 到 API 的接口修改矩阵

| 类别 | 路径/接口 | 是否改路径 | 需要修改的内容 | 优先级 |

## 4. 通用契约规则

## 4.1 鉴权

### 用户侧

1. 继续依赖浏览器登录态
2. 不对外开放
3. 不通过 query/header 传 provider token

### 任务侧

```http
X-API-Key: YOUR_API_KEY
Content-Type: application/json
```

规则：

1. 继续沿用当前 `/api/external/*` 的 API Key 体系
2. temp mailbox 读取授权由 `consumer_key` 归属控制
3. 底层 provider 的鉴权信息不得透出给调用方

## 4.2 时间字段

继续统一使用 ISO 8601：

```text
2026-03-27T12:00:00Z
```

## 4.3 错误结构

用户侧继续优先使用：

```json
{
  "success": false,
  "error": {
    "code": "ERROR_CODE",
    "message": "错误说明"
  }
}
```

任务侧继续优先使用：

```json
{
  "success": false,
  "code": "ERROR_CODE",
  "message": "错误说明",
  "data": {}
}
```

规则：

1. 不直接透出上游 provider 原始异常文本
2. 不把 provider HTTP 状态码直接当成正式产品错误码
3. finish、probe、wait 的终态错误码必须稳定

## 4.4 静态路由优先级

以下静态路径必须优先于动态邮箱路径注册：

1. `/api/temp-emails/options`
2. `/api/temp-emails/{email_addr}/extract-verification`

避免被 `/api/temp-emails/<path:email_addr>` 之类的动态路由误吞。

## 5. 用户侧正式接口

## 5.1 `GET /api/temp-emails/options`

```http
GET /api/temp-emails/options
需要登录：是
```

用途：

1. 告诉前端是否允许多域名选择
2. 告诉前端默认创建模式
3. 告诉前端前缀规则
4. 告诉前端当前 temp-mail 正式 provider 语义

成功响应示例：

```json
{
  "success": true,
  "options": {
    "provider": "temp_mail",
    "domain_strategy": "auto_or_manual",
    "default_mode": "auto",
    "domains": [
      {
        "name": "mail.example.com",
        "is_default": true,
        "enabled": true
      },
      {
        "name": "temp.example.net",
        "is_default": false,
        "enabled": true
      }
    ],
    "prefix_rules": {
      "min_length": 1,
      "max_length": 32,
      "pattern": "^[a-z0-9][a-z0-9._-]*$"
    }
  }
}
```

二期要求修改点：

1. `provider` 应表示正式 temp-mail 能力，不应直接暴露 legacy `gptmail`
2. `domains` 必须来自当前 runtime 真正可用域名，而不是硬编码
3. 若 options 加载失败，必须返回明确错误，例如 `TEMP_MAIL_OPTIONS_UNAVAILABLE`
4. 前端后续只允许依赖本接口，不允许改为直接请求底层 provider

## 5.3 `POST /api/temp-emails/generate`

```http
POST /api/temp-emails/generate
需要登录：是
Content-Type: application/json
```

请求体：

```json
{
  "prefix": "demo123",
  "domain": "mail.example.com"
}
```

请求规则：

1. `prefix` 可选
2. `domain` 可选
3. `domain` 为空表示自动分配
4. 不允许额外传入 provider-specific 参数

成功响应示例：

```json
{
  "success": true,
  "email": "demo123@mail.example.com",
  "mailbox": {
    "email": "demo123@mail.example.com",
    "prefix": "demo123",
    "domain": "mail.example.com",
    "source": "custom_domain_temp_mail",
    "mailbox_type": "user",

    "status": "active",
    "created_at": "2026-03-27T12:01:00Z"

  "message": "临时邮箱创建成功"
}
```

二期要求修改点：

1. controller 不再直接调用具体 provider，而只调用 `TempMailService`
2. 创建成功后，本地必须已形成完整 mailbox record
3. mailbox record 至少要包含 `source/prefix/domain/visible_in_ui/status/meta_json`
4. 响应继续只返回正式 mailbox 字段，不回显 provider handle

## 8. `/api/settings` 正式契约

二期不新增新的 settings 路径，继续复用 `/api/settings`。

但为了满足 FD 中“settings 与 runtime 语义一致”，本接口必须正式承载 temp-mail 平台配置。

## 8.1 `GET /api/settings`

temp-mail 相关正式返回字段：

| 字段 | 类型 | 必需 | 说明 |
|------|------|------|------|
| `temp_mail_provider` | string | 是 | 当前正式 temp-mail provider 类型 |
| `temp_mail_api_base_url` | string | 否 | provider 基础地址 |
| `temp_mail_api_key_set` | boolean | 是 | 是否已配置 API Key |
| `temp_mail_api_key_masked` | string | 否 | 脱敏后的 API Key |
| `temp_mail_domains` | array | 是 | 当前可用域名列表 |
| `temp_mail_default_domain` | string | 否 | 默认域名 |
| `temp_mail_prefix_rules` | object | 是 | 前缀规则 |

说明：

1. `gptmail_api_key` 不再作为正式返回字段给前端
2. temp-mail 设置页只应消费 `temp_mail_*`
3. 可兼容旧配置，但不能再把旧字段当正式产品语义展示

## 8.2 `PUT /api/settings`

建议接收字段：

```json
{
  "temp_mail_provider": "custom_domain_temp_mail",
  "temp_mail_api_base_url": "https://temp-mail.example.com/api",
  "temp_mail_api_key": "sk_live_xxx",
  "temp_mail_domains": [
    "mail.example.com",
    "temp.example.net"
  ],
  "temp_mail_default_domain": "mail.example.com",
  "temp_mail_prefix_rules": {
    "min_length": 1,
    "max_length": 32,
    "pattern": "^[a-z0-9][a-z0-9._-]*$"
  }
}
```

二期要求修改点：

1. `temp_mail_*` 成为唯一正式写入契约
2. legacy `gptmail_api_key` 只允许做兼容迁移，不允许反向污染正式字段
3. 空字符串 `temp_mail_api_key` 默认表示“不修改”，不能误清空已存在配置
4. 若后续需要显式清空 API Key，应单独设计 clear 语义，而不是复用空字符串

## 9. 内部平台化接口

本节不是新增 HTTP 路径，而是二期必须补齐的内部接口契约。

如果这些内部接口不成型，FD 就无法真正落地。

## 9.1 Mailbox Descriptor

内部统一 mailbox descriptor 建议最少包含：

```json
{
  "email": "demo123@mail.example.com",
  "mailbox_type": "user",
  "visible_in_ui": true,
  "status": "active",
  "source": "custom_domain_temp_mail",
  "prefix": "demo123",
  "domain": "mail.example.com",
  "task_token": "",
  "consumer_key": "",
  "caller_id": "",
  "task_id": "",
  "meta": {}
}
```

规则：

1. 这是内部模型，不要求原样对外返回
2. `meta` 用于存 provider-specific handle
3. 上层 controller、external 和 extractor 都应基于 mailbox descriptor 工作

## 9.2 Provider Base Contract

目标接口应从“按 `email_addr` 传参”升级为“按 mailbox 传参”：

```python
class TempMailProviderBase:
    def get_options(self) -> dict: ...
    def create_mailbox(self, *, prefix: str | None = None, domain: str | None = None) -> dict: ...
    def delete_mailbox(self, mailbox: dict) -> bool: ...
    def list_messages(self, mailbox: dict) -> list[dict] | None: ...
    def get_message_detail(self, mailbox: dict, message_id: str) -> dict | None: ...
    def delete_message(self, mailbox: dict, message_id: str) -> bool: ...
    def clear_messages(self, mailbox: dict) -> bool: ...
```

说明：

1. 当前 `generate_mailbox(...)` 可作为兼容别名短期保留
2. 目标契约应统一为 `create_mailbox(...)`
3. 之后 service 不再直接拼装 `email_addr` 作为唯一上下文

## 9.3 Provider Factory

二期建议新增内部工厂接口：

```python
def get_temp_mail_provider(provider_name: str | None = None) -> TempMailProviderBase: ...
```

职责：

1. 根据 settings/runtime 返回 provider 实例
2. 兼容 legacy bridge，但不把 legacy bridge 当正式产品能力暴露
3. 让 `TempMailService` 不再默认硬编码 `CustomTempMailProvider()`

## 9.4 TempMailService

二期目标语义：

1. 先按邮箱地址或 task_token 取 mailbox record
2. 再通过 provider factory 拿 provider
3. 再把 mailbox descriptor 传入 provider
4. 最后由 service 统一输出主项目自己的 message / verification 语义

换句话说，service 应该成为“业务语义层”，而不是 provider 直通层。

## 12. 一句话结论

二期 API 真正要改的，不是再扩一组新路径，而是把现有 `/api/temp-emails/*`、`/api/external/*`、`/api/settings` 这些正式接口的字段和行为契约稳定下来，同时补齐 mailbox descriptor、provider base、provider factory、service 调用边界这四个内部接口层，让 FD 所要求的“上层稳定、底层可替换”真正可实施。
