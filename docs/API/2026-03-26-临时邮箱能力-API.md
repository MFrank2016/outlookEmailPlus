- 文档版本: v0.2
- 创建日期: 2026-03-26
- 最近更新: 2026-03-27
- 文档类型: 目标接口契约
- 关联 PRD: `docs/PRD/2026-03-26-临时邮箱能力-功能PRD.md`
- 关联 FD: `docs/FD/2026-03-26-临时邮箱能力-FD.md`
- 当前范围: 仅定义临时邮箱能力的接口行为、字段结构、鉴权方式与改造范围

## 1. 文档定位

本文档描述的是“临时邮箱能力”的目标 API 契约，并同步记录当前主线实现已经收口到的正式行为，用于指导后续前端联调、后端实现和注册机接入。

需要明确：

1. 本文档不是当前代码的完整现状说明。
2. 当前正式实现已切换到“自建域名临时邮箱能力”语义。
3. `GPTMail` 不再作为正式产品能力的一部分，仅保留内部兼容 bridge。
4. 文档末尾会明确列出当前仍保留的兼容项，而不是把 legacy bridge 当正式契约的一部分。

## 2. 接口分层

临时邮箱能力分成两套接口：

### 2.1 用户侧接口

面向项目 Web 前端使用。

特点：

1. 依赖浏览器登录态
2. 只返回用户手动创建的可见邮箱
3. 接口风格尽量延续当前 `/api/temp-emails/*`
4. 前端无需感知底层是哪个临时邮箱来源

### 2.2 任务侧接口

面向注册机 / 自动化调用方使用。

特点：

1. 使用 `X-API-Key`
2. 不依赖浏览器登录态
3. 任务邮箱默认不进入前端列表
4. 只新增最小必要接口，避免重复建设第二套读信协议

## 3. 通用规则

### 3.1 用户侧鉴权

用户侧接口继续使用当前登录态校验：

1. 需要已登录
2. 不对外开放
3. 返回风格延续当前 Web API 结构

### 3.2 任务侧鉴权

任务侧接口统一要求：

```http
X-API-Key: YOUR_API_KEY
Content-Type: application/json
```

说明：

1. 继续沿用当前 `/api/external/*` 的 API Key 体系
2. 不依赖 Cookie
3. 不依赖 CSRF Token
4. 默认返回统一结构：`success + code + message + data`

### 3.3 时间字段

所有时间字段统一使用 ISO 8601，例如：

```text
2026-03-26T12:00:00Z
```

### 3.4 多域名规则

采用方案 B：

1. 默认自动分配域名
2. 支持显式指定域名
3. 指定域名不可用时返回明确错误

### 3.5 前缀规则

1. `prefix` 为可选字段
2. 不传时系统自动生成
3. 传入时系统按规则校验并尝试创建
4. 前缀冲突或格式非法时返回明确错误

### 3.6 设置项规则

临时邮箱能力切换后，设置页与后端接口之间也需要明确契约。

本期要求：

1. 后台设置页读取和保存的正式字段应切到 `temp_mail_*`
2. 旧 `gptmail_api_key` 只保留兼容读取 / 兼容迁移，不再作为正式产品字段暴露
3. 设置接口返回时，敏感值继续采用 masked / is_set 形式，不直接回显明文 API Key

建议纳入当前 `/api/settings` 契约的字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `temp_mail_provider` | string | 当前临时邮箱 provider 类型 |
| `temp_mail_api_base_url` | string | 上游临时邮箱服务地址 |
| `temp_mail_api_key_set` | boolean | 是否已配置 API Key |
| `temp_mail_api_key_masked` | string | 脱敏后的 API Key |
| `temp_mail_domains` | array | 可用域名列表 |
| `temp_mail_default_domain` | string | 默认域名 |
| `temp_mail_prefix_rules` | object | 前缀规则 |

说明：

1. 本文档不另起一套新的 settings 路径
2. 但临时邮箱能力相关设置字段必须进入现有 settings 接口契约

## 4. 用户侧接口

## 4.1 获取临时邮箱配置项

```http
GET /api/temp-emails/options
需要登录：是
```

用途：

1. 让前端知道是否有多个可用域名
2. 决定是否展示“域名下拉选择”
3. 告知默认是否自动分配

成功响应示例：

```json
{
  "success": true,
  "options": {
    "domain_strategy": "auto_or_manual",
    "default_mode": "auto",
    "domains": [
      {
        "name": "mail.example.com",
        "is_default": true,
        "enabled": true
      },
      {
        "name": "temp.example.net",
        "is_default": false,
        "enabled": true
      }
    ],
    "prefix_rules": {
      "min_length": 1,
      "max_length": 32,
      "pattern": "^[a-z0-9][a-z0-9._-]*$"
    }
  }
}
```

说明：

1. 当前项目中该接口尚不存在
2. 这是支持前端域名选择所必需的新接口

## 4.2 获取用户可见临时邮箱列表

```http
GET /api/temp-emails
需要登录：是
```

用途：

1. 获取当前用户侧可见邮箱列表
2. 不返回 API / 注册机创建的任务邮箱

成功响应示例：

```json
{
  "success": true,
  "emails": [
    {
      "email": "demo123@mail.example.com",
      "prefix": "demo123",
      "domain": "mail.example.com",
      "source": "custom_domain_temp_mail",
      "created_by": "user",
      "visible_in_ui": true,
      "created_at": "2026-03-26T12:00:00Z"
    }
  ]
}
```

字段说明：

| 字段 | 类型 | 说明 |
|------|------|------|
| `email` | string | 完整邮箱地址 |
| `prefix` | string | 邮箱前缀 |
| `domain` | string | 域名 |
| `source` | string | 当前来源标识 |
| `created_by` | string | 固定为 `user` |
| `visible_in_ui` | boolean | 固定为 `true` |
| `created_at` | string | 创建时间 |

## 5. 任务侧接口

任务侧接口采用“最小新增”原则。

设计原则：

1. 任务邮箱不进入前端列表
2. 调用方不直接依赖底层临时邮箱项目
3. 已有 external 读信接口尽量复用
4. 只为“申请邮箱”和“结束使用”新增接口

## 5.1 复用现有 external 读取接口

任务邮箱申请成功后，注册机继续复用现有对外接口：

1. `GET /api/external/messages`
2. `GET /api/external/messages/latest`
3. `GET /api/external/messages/{message_id}`
4. `GET /api/external/verification-code`
5. `GET /api/external/verification-link`
6. `GET /api/external/wait-message`

复用方式：

1. `apply` 接口返回任务邮箱地址
2. 调用方后续直接把该邮箱地址传给现有 external 读取接口
3. 因此不再额外设计 `/api/external/temp-emails/{task_token}/messages*` 这一整套重复接口

前提要求：

1. external 读取服务层必须能够识别并读取任务邮箱
2. 不能只支持 `accounts` 表中的普通邮箱
3. 需要把“普通邮箱读取”和“临时邮箱读取”统一到同一套服务入口
## 5.2 申请任务邮箱

## 5.4 finish 与异步等待边界规则

任务邮箱申请成功后，调用方可能还会使用：

1. `GET /api/external/wait-message`
2. `GET /api/external/probe/{probe_id}`

因此需要补充以下边界规则：

1. `finish` 成功后，新的 `/api/external/messages*`、`/api/external/verification-*`、`/api/external/wait-message` 请求默认应被拒绝
2. 如果 `finish` 发生时已经存在进行中的 `wait-message` 同步等待，建议立即结束并返回任务已结束错误
3. 如果 `finish` 发生时已经存在异步 `probe`，建议将 probe 状态转为已终止，而不是继续等待超时
4. 对已终止 probe 的查询结果，建议返回稳定错误码，避免调用方只能依赖字符串判断

建议错误码补充：

| 错误码 | HTTP 状态码 | 说明 |
|--------|-------------|------|
| `TASK_FINISHED` | 409 | 任务邮箱已结束，禁止继续读取 |
| `PROBE_CANCELLED` | 409 | 探测因任务结束而被终止 |

## 6. 错误码建议

### 6.1 通用错误

| 错误码 | HTTP 状态码 | 说明 |
|--------|-------------|------|
| `UNAUTHORIZED` | 401 | 缺少或无效的 API Key |
| `AUTH_REQUIRED` | 401 | 未登录 |
| `FORBIDDEN` | 403 | 无访问权限 |
| `INVALID_PARAM` | 400 | 参数无效 |
| `INTERNAL_ERROR` | 500 | 服务内部错误 |

### 6.2 临时邮箱能力错误

| 错误码 | HTTP 状态码 | 说明 |
|--------|-------------|------|
| `TEMP_EMAIL_CREATE_FAILED` | 502 | 创建邮箱失败 |
| `TEMP_EMAIL_EXISTS` | 409 | 邮箱已存在 |
| `TEMP_EMAIL_NOT_FOUND` | 404 | 邮箱不存在 |
| `DOMAIN_NOT_AVAILABLE` | 400 | 指定域名不可用 |
| `PREFIX_INVALID` | 400 | 前缀格式非法 |
| `PREFIX_CONFLICT` | 409 | 前缀冲突 |
| `TEMP_EMAIL_MESSAGE_NOT_FOUND` | 404 | 邮件不存在 |
| `VERIFICATION_CODE_NOT_FOUND` | 404 | 未匹配到验证码 |
| `VERIFICATION_LINK_NOT_FOUND` | 404 | 未匹配到验证链接 |
| `TASK_TOKEN_INVALID` | 404 | 任务令牌无效 |
| `TASK_ALREADY_FINISHED` | 409 | 任务已结束 |

## 7. 当前实现与目标契约差异

## 7.1 当前已存在的接口

当前代码里已经存在：

1. `GET /api/temp-emails`
2. `POST /api/temp-emails/generate`
3. `DELETE /api/temp-emails/{email_addr}`
4. `GET /api/temp-emails/{email_addr}/messages`
5. `GET /api/temp-emails/{email_addr}/messages/{message_id}`
6. `DELETE /api/temp-emails/{email_addr}/messages/{message_id}`
7. `DELETE /api/temp-emails/{email_addr}/clear`
8. `POST /api/temp-emails/{email_addr}/refresh`

## 7.2 当前已补齐的关键接口

当前项目已经补齐：

1. `GET /api/temp-emails/options`
2. `GET /api/temp-emails/{email_addr}/extract-verification`
3. `POST /api/external/temp-emails/apply`
4. `POST /api/external/temp-emails/{task_token}/finish`

## 7.3 当前实现与目标契约的剩余兼容项

### A. `gptmail.py` 仍保留为内部 bridge

当前项目已经存在：

1. `temp_mail_provider_custom.py`
2. `temp_mail_service.py`
3. `mailbox_resolver.py`

正式用户接口、external 接口、settings 契约和页面文案均已切到临时邮箱 / 自建临时邮箱语义。

当前仍保留的 `outlook_web.services.gptmail` 只作为内部兼容 bridge：

1. 继续承接 legacy 上游请求
2. 不再作为正式 provider/type/source 对外输出
3. 不再作为正式页面文案、settings 正式字段或导出标题的一部分

### B. `temp_emails` 与消息缓存模型已升级

当前 `temp_emails` 已具备以下字段以支持正式能力：

1. `source`
2. `visible_in_ui`
3. `prefix`
4. `domain`
5. `task_token`
6. `consumer_key`
7. `caller_id`
8. `task_id`
9. `finished_at`

`temp_email_messages` 也已按 `(email_address, message_id)` 作用域缓存，避免不同邮箱消息 ID 冲突。

### C. 前端已切到 temp-mail 正式创建与提取链路

当前前端已经支持：

1. 读取 `/api/temp-emails/options`
2. 创建时提交 `prefix + domain`
3. 临时邮箱提取走 `/api/temp-emails/{email_addr}/extract-verification`
4. temp-email 页面不再把 `GPTMail` 作为正式产品文案

### D. external 读取链路已支持任务邮箱

当前 `/api/external/messages*`、`/api/external/verification-*`、`/api/external/wait-message` 已可直接读取任务邮箱。

读取授权规则已经分流为：

1. 普通邮箱继续走 `allowed_emails`
2. 任务邮箱走 `consumer_key` 归属校验
3. `finish` 后继续读取返回稳定 `TASK_FINISHED`
4. 已存在 probe 在 `finish` 后转为 `cancelled / PROBE_CANCELLED`

### E. settings 正式契约已经收口到 `temp_mail_*`

当前 `/api/settings` 的正式字段已为：

1. `temp_mail_provider`
2. `temp_mail_api_base_url`
3. `temp_mail_api_key_set`
4. `temp_mail_api_key_masked`
5. `temp_mail_domains`
6. `temp_mail_default_domain`
7. `temp_mail_prefix_rules`

旧 `gptmail_api_key` 仅保留兼容读取与兼容迁移，不再作为正式返回字段暴露。

### F. 正式产品文案与对外语义已去 GPT 化

当前正式对外语义已经收口为：

1. `临时邮箱`
2. `temp mail`
3. `custom_domain_temp_mail`

不再作为正式产品能力出现的内容包括：

1. `GPTMail` 页面文案
2. `GPTMail 临时邮箱服务` 分组描述
3. `GPTMail API` 启动日志
4. 导入导出中的正式 `GPTMail` 标题或正式 provider 展示名

## 8. 本轮已完成的主线改动

1. 已为临时邮箱能力落地统一适配层，`controller -> temp_mail_service -> provider/repo`
2. 已补齐用户侧 `options` / `extract-verification` 和任务侧 `apply` / `finish`
3. 已完成 `/api/external/messages*` 对任务邮箱的 mailbox resolver 分流与授权兼容
4. 已将正式 settings 契约收口到 `temp_mail_*`
5. 已清理页面、设置页、导入导出和启动日志中的正式 `GPTMail` 产品语义
6. 已保留 `legacy_gptmail` / `gptmail_api_key` / `gptmail.py` 作为内部兼容层，而不是正式对外语义

## 9. 当前结论
