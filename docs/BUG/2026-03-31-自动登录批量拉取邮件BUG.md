# BUG 报告：自动登录时批量拉取邮件 + 邮件获取失败

**报告日期**：2026-03-31  
**修复完成日期**：2026-03-31  
**发现人**：用户手动验收  
**严重程度**：🔴 高  
**影响范围**：所有用户的首次登录体验  
**状态**：✅ 已修复

---

## 一、问题描述

### BUG-1：自动登录进入后强制为所有邮箱拉取邮件

**现象**：用户登录进入系统后，还没有手动选择任何账号，系统就自动对当前分组内**所有**邮箱账号发起了邮件拉取请求。

**预期行为**：首次登录进入时，只应加载分组和账号列表，**不应自动拉取邮件**。邮件拉取应在以下场景触发：
1. 用户**手动选择某个账号**时，加载该账号的邮件
2. 用户**手动开启轮询**后，按配置的间隔周期拉取

### BUG-2：邮件获取失败

**现象**：由于 BUG-1 导致的大量并发请求，部分或全部邮件获取失败（后端可能因为 Token 验证、API 限流等原因拒绝请求）。

**直接原因**：BUG-1 触发了 N 个账号 × 4 个请求 = 4N 个并发 API 调用，后端或微软 Graph API 无法承受这种突发负载。

## 二、根因分析

### 最终确认的调用链路（通过浏览器 console.trace 定位）

```
用户点击 Dashboard 分组列表
  → onclick: navigate('mailbox'); setTimeout(function(){ selectGroup(groupId); }, 100)
    → selectGroup(groupId)
      → loadAccountsByGroup(groupId, false, true)    // startPolling=true
        → forEach(accounts) → startPoll(acc.email)   // ⚠️ 批量轮询启动！
          → 每个 startPoll 发起 2 个 API 请求建立 baseline
          → 150ms 后再发 2 个请求做首次轮询
```

**或者**（首次进入时）：

```
DOMContentLoaded
  → loadGroups()
    → selectGroup(firstNormalGroup.id)
      → loadAccountsByGroup(groupId, false, true)    // startPolling=true
        → forEach(accounts) → startPoll(acc.email)   // ⚠️ 批量轮询启动！
```

### 问题代码位置

**文件**：`static/js/features/groups.js`

`loadAccountsByGroup` 中有**两处**批量轮询启动代码（CACHED 路径和 FETCH 路径）：
```javascript
// 标准模式：加载分组后，为所有账号启动轮询（跳过已在轮询中的）
var view = typeof mailboxViewMode !== 'undefined' ? mailboxViewMode : 'standard';
if (startPolling && view !== 'compact' && typeof pollEnabled !== 'undefined' && pollEnabled && typeof startPoll === 'function') {
    var pollCount2 = 0;
    (data.accounts || []).forEach(function(acc) {
        if (acc && acc.email && !(hasPollMap2 && pollMap.has(acc.email))) {
            startPoll(acc.email, { silent: true }); pollCount2++;
        }
    });
    if (pollCount2 > 0 && typeof showToast === 'function') {
        showToast(translateAppTextLocal('已为 ' + pollCount2 + ' 个账号启动轮询监听'), 'info');
    }
}
```

### 影响计算

如果一个分组有 50 个账号：
- 账号列表加载：1 个 API 请求
- 批量 startPoll：50 × 2 = 100 个 baseline 请求（并发）
- 150ms 后首次轮询：50 × 2 = 100 个轮询请求
- **总计：201 个 API 请求在 1 秒内发出**

## 三、修复过程与踩坑记录

### 第一次修复尝试（失败）

**方案**：添加 `_initialLoadComplete` 标志位，首次加载时为 `false`，加载完成后设为 `true`。批量轮询条件改为 `_initialLoadComplete && ...`。

**结果**：**失败**。因为 `loadGroups()` 在 `DOMContentLoaded` 时就被调用（`main.js:948`），`_initialLoadComplete` 在 Dashboard 加载后就变成了 `true`。当用户后来 `navigate('mailbox')` 时，标志位已经是 `true`，仍然触发批量轮询。

**教训**：全局标志位在 SPA（单页面应用）中容易被提前触发，不可靠。

### 第二次修复尝试（部分成功）

**方案**：改用 `loadAccountsByGroup` 的第三参数 `startPolling`（默认 `false`）。只有 `selectGroup` 传 `startPolling=true`。

**结果**：**部分成功**。修复了 `navigate('mailbox')` → `loadAccountsByGroup(currentGroupId)` 的路径（默认 `startPolling=false`），但 `selectGroup` 仍然传 `true`。

**遗漏路径**：首次进入时 `loadGroups()` → `selectGroup(firstNormalGroup.id)` → `loadAccountsByGroup(groupId, false, true)` 仍然触发批量轮询。

### 第三次修复尝试（仍不够）

**方案**：给 `selectGroup` 添加 `autoStartPolling` 参数（默认 `true`），首次自动选中时传 `false`。

**结果**：**不够**。因为还有其他路径调用 `selectGroup` 且不传参数（默认 `true`）：
- `main.js:601`：Dashboard 页面点击分组列表 → `navigate('mailbox'); setTimeout(function(){ selectGroup(groupId); }, 100)` — 只传了 `groupId`，`autoStartPolling` 默认 `true`
- 用户从浏览器调用栈确认了这正是触发批量轮询的路径

**教训**：参数默认值策略要考虑**所有调用者**，特别是 HTML 内联 `onclick` 中的调用（无法传额外参数）。

### 第四次修复（最终成功）

**方案**：**彻底移除** `loadAccountsByGroup` 中的两处批量轮询逻辑。轮询只通过 `selectAccount`（用户手动选中单个账号）触发。

**结果**：✅ **成功**。批量轮询问题彻底解决。

## 四、最终修复方案

### 核心原则

**轮询只在用户选中单个账号时启动**，不在加载分组/切换分组时批量启动。

### 代码变更

1. **`groups.js`**：删除 `loadAccountsByGroup` 中的两处批量轮询代码块（CACHED 路径和 FETCH 路径），共约 24 行
2. **`groups.js`**：移除 `startPolling` 参数和 `autoStartPolling` 参数（不再需要）
3. **`groups.js`**：删除 `_initialLoadComplete` 全局变量
4. **`accounts.js`**：`selectAccount` 中保留单个账号的轮询启动逻辑（这是正确的行为）
5. **`mailbox_compact.js`**：`email-copied` 事件处理保留（复制邮箱后启动轮询，正确行为）

### 轮询触发点（修复后）

| 触发场景 | 轮询行为 | 位置 |
|----------|----------|------|
| 用户选中单个账号 | 启动该账号的轮询 | `accounts.js:selectAccount` |
| 用户复制邮箱地址 | 启动该邮箱的轮询 | `mailbox_compact.js:email-copied` |
| 加载分组/切换分组 | ❌ 不启动轮询 | 已移除 |
| 首次登录进入 | ❌ 不启动轮询 | 已移除 |
| 导航到邮箱页面 | ❌ 不启动轮询 | 已移除 |

## 五、经验总结

### 1. SPA 中全局标志位不可靠

在单页面应用中，`DOMContentLoaded` 只触发一次，之后页面切换都是通过 `navigate()` 函数实现。使用全局标志位（如 `_initialLoadComplete`）来区分"首次加载"和"后续操作"非常不可靠，因为：
- 标志位可能被其他页面（如 Dashboard）提前设置
- 多个异步流程可能竞态修改同一个标志

**正确做法**：通过函数参数传递意图，或者直接从调用方决定是否执行某个行为。

### 2. 默认参数值要考虑所有调用者

给函数添加参数（如 `startPolling=false`）时，需要检查**所有调用者**，包括：
- JS 代码中的直接调用
- HTML 内联 `onclick` 中的调用（无法传额外参数）
- `setTimeout`/`setInterval` 中的调用
- 动态生成的 HTML 字符串中的调用

**本次踩坑**：`main.js:601` 中 Dashboard 的分组点击使用了内联 `onclick`：
```javascript
`navigate('mailbox'); setTimeout(function(){ selectGroup(${g.id}); }, 100)`
```
这里 `selectGroup` 只传了 `groupId`，新增的 `autoStartPolling` 参数默认 `true`，仍然触发批量轮询。

### 3. "渐进式修复"不如"根本性修复"

前三次修复尝试都是在保留批量轮询逻辑的前提下，试图通过条件判断来控制触发时机。但由于调用路径太多、条件组合复杂，总是有遗漏。

**最终成功的方案是彻底移除批量轮询逻辑**——如果一个功能在任何场景下都不应该触发，就不应该保留它的代码并依赖条件守卫。

### 4. 浏览器缓存是调试的隐形障碍

修改 JS 文件后，如果浏览器加载的仍然是缓存版本，修复不会生效。本项目使用 `?v={{ APP_VERSION }}` 做 cache busting，但版本号需要手动更新。

**解决方案**：修改 JS 后同步升级 `outlook_web/__init__.py` 中的 `__version__`（从 `1.10.1` → `1.10.2`），强制浏览器加载新文件。

### 5. console.trace 是定位调用链的利器

在排查"谁调用了这个函数"时，`console.trace()` 比 `console.log()` 更有价值——它会打印完整的调用栈，直接显示调用链路：
```javascript
function startPoll(email, opts) {
    console.trace('[poll-engine] startPoll call stack for:', email);
    // ...
}
```

本次正是通过 `console.trace` 定位到 `selectGroup` → `loadAccountsByGroup` 的真实调用路径。
