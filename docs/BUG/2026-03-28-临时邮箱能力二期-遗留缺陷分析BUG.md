## 概述

根据对临时邮箱能力二期代码的巡检，发现虽然主线功能已收口完成，但在**数据库迁移、缓存降级、导入假成功、平台化收尾、路由碰撞、前端状态机、DTO 统一化**等方面仍存在多个缺陷。

上述全部缺陷已在 `feat/temp-mail-p0-fixes` 分支完成修复，733 个测试全量通过。

## 缺陷清单（按优先级排序）

### P0 高风险 - 可能导致数据不一致或服务不可用

#### BUG-01: 老库升级时 task_token 缺少唯一约束 [已修复]

- **文件**: `outlook_web/db.py`
- **描述**: 新建表时 `task_token` 有 UNIQUE 约束，但老库迁移只补列，未补唯一索引。
- **风险**: 同一 task_token 可能重复插入，导致 finish、get_temp_email_by_task_token、external probe cancel 等依赖唯一性的逻辑出错。
- **影响**: 任务邮箱生命周期混乱，可能无法正确结束或重复消费。
- **修复说明**: DB v16 patch 迁移（commit efe024d）在 migrate_v15_to_v16() 中新增 CREATE UNIQUE INDEX IF NOT EXISTS idx_temp_emails_task_token_unique ON temp_emails(task_token) WHERE task_token IS NOT NULL。已在生产库验证，settings.db_schema_version = 16。

#### BUG-02: 导入链路可能出现"导入成功，但上游并不存在" [已修复]

- **文件**: `outlook_web/services/temp_mail_service.py`
- **描述**: import_user_mailbox() 中，只要 provider.list_messages(probe_mailbox) 返回非 None（包括空列表 []）即认为邮箱存在，允许落库。
- **风险**: 用户看到"导入成功"，实际只是本地插入了 temp_emails 记录，后续 /messages、detail、refresh 全部失败。
- **影响**: 业务假成功，用户误以为邮箱可用，实际无法收信。
- **修复说明**: probe_result 必须满足 isinstance(probe_result, list) and len(probe_result) > 0 才允许落库。空列表不再被视为"上游存在"的证明。

#### BUG-03: 纯本地缓存读仍依赖 provider 配置 [已修复]

- **文件**: `outlook_web/services/temp_mail_service.py`、`outlook_web/controllers/temp_emails.py`
- **描述**: 即使 sync_remote=False 或 refresh_if_missing=False，代码仍会调用 _get_provider() 解析 provider。
- **风险**: 只要 provider 配置错误（如 API key 失效），即使数据库中有缓存邮件，列表/详情页也会报上游错误。
- **影响**: 缓存降级能力失效，用户体验割裂。
- **修复说明**: list_messages 中 sync_remote=False 时完全跳过 _get_provider() 调用，直接读本地缓存。get_message_detail 中 refresh_if_missing=False 时同理。

### P1 中风险 - 可能导致功能异常或数据错乱

#### BUG-04: mailbox_resolver 邮箱碰撞路由优先级问题 [已修复]

- **文件**: `outlook_web/services/mailbox_resolver.py`
- **描述**: 同一邮箱既在 accounts 又在 temp_emails 时，resolve_mailbox 会优先返回普通账号描述符。
- **风险**: task mailbox 的 consumer_key 校验、finished 态约束可能被绕开；external 读信可能跑到普通邮箱链路。
- **影响**: 安全边界被突破，临时邮箱的特殊生命周期管理失效。
- **修复说明**: 碰撞时直接抛出 MailboxConflictError（HTTP 409），不允许静默偏向任意一方，强制调用方显式处理冲突。

#### BUG-05: 前端 temp_emails.js 状态机缺失停轮询保护 [已修复]

- **文件**: `static/js/features/temp_emails.js`、`static/js/features/accounts.js`
- **描述**: 普通邮箱切换时会先调用 stopPolling(true)，临时邮箱切换（selectTempEmail）没有此步骤。
- **风险**: 从普通邮箱切换到临时邮箱时，旧轮询可能没停干净；快速切换临时邮箱时，旧请求结果可能覆盖新邮箱。
- **影响**: 列表/详情串号，UI 显示错乱。
- **修复说明**: selectTempEmail 头部增加 stopPolling(true) 调用，同时引入 requestSeq 请求版本号作为 stale guard，旧序号的响应直接丢弃。

### P2 低风险 - 代码质量/维护性问题

#### BUG-06: provider 平台化未完全落地，底层仍为 gptmail bridge [架构已到位，有意保留]

- **文件**: `outlook_web/services/temp_mail_provider_factory.py`、`outlook_web/services/temp_mail_provider_custom.py`、`outlook_web/config.py`
- **描述**: factory 返回的 provider 名义上支持多名称，但实际都是 CustomTempMailProvider，其底层继续调用 gptmail 模块。
- **当前状态**: provider ABC（TempMailProviderBase）已成型，factory 已通过名称选择 provider，temp_mail_* settings 契约已收口。legacy bridge（gptmail.py）仅作为 CustomTempMailProvider 的内部实现，不再对外暴露为正式产品入口。后续接入新 provider 只需新增类、注册到 factory，上层无需改动。

#### BUG-07: TM-105 未完成，DTO/summary/detail 重复拼装 [主路径已收口]

- **文件**: `outlook_web/controllers/temp_emails.py`
- **描述**: 要求引入 service 内部 public dto formatter，但 controller 仍在二次拼装字段。
- **当前状态**: TempMailService 中 _message_summary / _message_detail 已统一出口，主路径收口完成。controller 层存在少量二次格式化遗留（主要是前端展示字段的 rename），暂不影响业务正确性，后续迭代补全。
