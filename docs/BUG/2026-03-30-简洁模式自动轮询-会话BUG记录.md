## 已修复的 Bug

### BUG-1：`.mail-row` 缺少 `data-email` 属性（阻塞）

**现象**：`findCompactAccountRow(email)` 始终返回 `null`，轮询一触发就报"页面元素丢失，已停止监听"。

**根因**：`renderCompactAccountList` 中 `.mail-row` div 没有 `data-email` 属性，选择器 `.mail-row[data-email="..."]` 匹配不到任何元素。

**修复**：在 `mailbox_compact.js` 第 240 行的 `.mail-row` 模板字符串中添加：
```html
data-email="${escapeHtml(account.email || '')}"
```

**提交**：`22fcdb3`

---

### BUG-2：简洁模式设置面板被移除（功能退化）

**现象**：用户发现简洁模式没有独立的设置面板（间隔、次数），无法单独配置。

**根因**：上一个对话的 commit `5393e23` 将简洁模式轮询合并到标准轮询（共用一套开关+间隔），删除了简洁模式专属面板。

**修复**：恢复简洁模式独立面板，将 `maxDuration（时间语义）`完全替换为 `maxCount（次数语义）`：
- `index.html`：恢复简洁模式面板（`enableCompactAutoPoll`、`compactPollInterval`、`compactPollMaxCount`）
- `mailbox_compact.js`：引擎改为按次数计数停止，`state.pollCount` 跟踪已执行次数
- `main.js`：独立读取/保存 compact poll 设置
- `settings.py`：字段名 `compact_poll_max_duration` → `compact_poll_max_count`，范围 0-100

**提交**：`22fcdb3`

### BUG-3：标准模式轮询指示器与 Toast 通知重叠

**现象**：标准模式下开启轮询后，右下角绿色悬浮框与 Toast 通知叠放，视觉混乱。

**根因**：`pollingStatusIndicator` 的 CSS 定位为 `bottom: 24px; right: 24px`，与 Toast 容器（`bottom: 1.5rem; right: 1.5rem`）完全重叠。

**修复（commit `da73ab5`）**：临时修改为左下角（`left: 24px`）。

**进一步修复（commit `b4c62db`）**：彻底改为在账号卡片邮箱地址旁插入小绿点（`.standard-poll-dot`），与 Toast 完全分离，与简洁模式风格统一。

### BUG-4：简洁模式轮询设置重启后不生效

**现象**：重启应用后，compact poll 功能不会自动启用，必须手动打开设置面板并保存一次才会生效。

**根因**：`initPollingSettings()`（DOMContentLoaded 时调用）只初始化了标准轮询（`applyPollingSettings`），没有初始化 compact poll 的 `compactPollEnabled/compactPollInterval/compactPollMaxCount` 变量，也没有调用 `applyCompactPollSettings`。

**修复**：在 `initPollingSettings()` 中补充 compact poll 的初始化逻辑：
```js
compactPollEnabled  = isAutoPollingEnabledSetting(data.settings.enable_compact_auto_poll);
compactPollInterval = parseIntegerSetting(data.settings.compact_poll_interval, 10);
compactPollMaxCount = parseIntegerSetting(data.settings.compact_poll_max_count, 5);
if (typeof applyCompactPollSettings === 'function') {
    applyCompactPollSettings({ enabled: compactPollEnabled, interval: compactPollInterval, maxCount: compactPollMaxCount });
}
```

**提交**：`b4c62db`

### BUG-5："页面元素丢失"假报错

**现象**：简洁模式轮询运行中，有时突然报"页面元素丢失，已停止监听"，即使账号实际存在。

**根因**：
1. 列表重新渲染时 DOM 被清空再重建，在渲染过程中触发了 `pollSingleEmail`，此时 `findCompactAccountRow` 找不到还未渲染完的行，就触发了停止逻辑
2. 从简洁模式切换到标准模式时，`.mail-row` DOM 被隐藏，同样导致找不到行

**修复**：
- 调换 DOM 检查和账号存在性检查的顺序（先检查账号是否在可见列表中）
- DOM 找不到但账号存在时，改为跳过本次 poll（而非停止轮询）
- 切换离开简洁模式时主动停止所有简洁轮询

**提交**：`da73ab5`

## 遗留 Bug（已全部修复）

### BUG-6：保存设置时弹出"简洁模式轮询次数必须在 0-100 之间"校验错误 ✅ 已修复

**现象**：用户在保存设置时，弹出提示"简洁模式轮询次数必须在 0-100 之间"的报错，但用户实际上没有填写该字段（或该字段值为空）。

**确认根因**：
`saveSettings` 函数**无条件**读取 compact poll 输入框并做严格校验（`isNaN → 阻断保存`），
而用户可能清空了输入框（`parseInt("") = NaN`）或根本没触碰这些字段。
与 Telegram 设置的处理方式不一致（Telegram 用 `if (!isNaN)` 跳过无效值）。

**修复**：
将 NaN 回退到内存中的当前设置值（`compactPollInterval` / `compactPollMaxCount`），不再阻断保存：
```js
const cPollIntervalVal = isNaN(cPollIntervalRaw) ? compactPollInterval : cPollIntervalRaw;
const cPollMaxCountVal = isNaN(cPollMaxCountRaw) ? compactPollMaxCount  : cPollMaxCountRaw;
// 只校验范围，不再因 NaN 阻断
if (cPollIntervalVal < 3 || cPollIntervalVal > 60) { ... }
if (cPollMaxCountVal < 0 || cPollMaxCountVal > 100) { ... }
```

**提交**：2026-03-31 修复

### BUG-7：标准模式轮询小绿点在账号列表重渲染后可能有延迟恢复 ✅ 已分析（无需修复）

**现象**：（未完全测试）`renderAccountList` 渲染完成后调用 `showPollingStatusIndicator()`，可能有短暂的"绿点消失"闪烁。

**分析结论**：
`renderAccountList` 是同步函数（`innerHTML` 赋值 + 立即调用 `showPollingStatusIndicator`），
不存在异步间隙。`.account-email` 的 `textContent.trim()` 能正确匹配 `currentAccount`。
绿点空 span 的 textContent 为 `""`，不影响 trim 匹配。
此问题为理论风险，实际不会发生。

**影响文件**：`static/js/features/groups.js`，`static/js/main.js`
