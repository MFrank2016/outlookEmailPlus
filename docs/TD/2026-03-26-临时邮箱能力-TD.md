## 1. 文档目标

本文档只回答技术实现问题，不重复解释产品需求。

本 TD 重点解决：

1. 如何把当前 `GPTMail` 强耦合实现改造成可替换的临时邮箱能力
2. 如何让用户侧继续沿用现有页面与接口路径
3. 如何让注册机侧复用现有 `/api/external/messages*` 读取链路
4. 如何用最小改动支持“用户邮箱可见、任务邮箱隐藏”
5. 如何在不引入复杂状态机的前提下完成 MVP

## 2. 当前技术现状

当前代码中的临时邮箱能力存在以下事实：

1. `outlook_web/controllers/temp_emails.py` 直接依赖 `outlook_web/services/gptmail.py`
2. `temp_emails` 表仅保存邮箱地址与简单状态
3. `temp_email_messages` 作为本地缓存表存在，但只服务于当前 GPTMail 链路
4. 前端 `static/js/features/temp_emails.js` 创建邮箱时不传 `prefix` / `domain`
5. 前端临时邮箱验证码提取错误复用了普通邮箱接口 `/api/emails/{email}/extract-verification`
6. `/api/external/messages*` 读取服务当前只认 `accounts` 表中的普通邮箱
7. 外部 API 的邮箱范围校验当前依赖 `allowed_emails` 静态名单，不适合动态创建的任务邮箱
8. 前端 temp-email 卡片虽然单独存在，但验证码提取实际仍走共享函数 `copyVerificationInfo()`，真实请求路径还是普通邮箱接口
9. temp-email 路由当前只有 `/api/temp-emails/<path:email_addr>` 这一类动态路径，如果新增 `/api/temp-emails/options` 时不处理注册顺序或匹配优先级，`options` 会被当成邮箱地址吃掉

这意味着本次不是单点替换，而是一次“小规模能力重构”。

## 3. 技术目标

## 3.1 本期必须实现

1. 用统一适配层替换 `gptmail.py` 直连模式
2. 支持用户邮箱与任务邮箱两类记录
3. 用户侧继续使用 `/api/temp-emails/*`
4. 外部侧仅新增 `apply` / `finish`
5. `/api/external/messages*` 可直接读取任务邮箱
6. 提取验证码 / 链接逻辑在用户侧和外部侧保持同一套结果定义

## 3.2 本期明确不做

1. 复杂生命周期状态机
2. 多租户用户隔离
3. 高级回收调度器
4. Cloudflare 自动开域名 / 自动建 DNS
5. 面向多 provider 的完整插件市场式架构

## 4. 核心技术决策

## 4.1 决策一：引入临时邮箱适配层，不再让 Controller 直接碰上游

### 结论

新增统一的临时邮箱服务分层：

```text
routes
  -> controllers
    -> temp_mail_service
      -> temp_mail_provider
        -> 上游临时邮箱系统
      -> temp_emails_repo / temp_mail_messages_repo
```

### 原因

当前 `controllers/temp_emails.py` 直接调用 `gptmail.py`，问题很明显：

1. Controller 和上游协议耦合过深
2. 无法平滑替换到自建域名临时邮箱来源
3. 无法同时服务用户侧和外部侧读取场景

### 建议新增模块

1. `outlook_web/services/temp_mail_service.py`
2. `outlook_web/services/temp_mail_provider_base.py`
3. `outlook_web/services/temp_mail_provider_custom.py`
4. `outlook_web/services/mailbox_resolver.py`

### 适配层接口建议

```python
class TempMailProvider:
    def get_options(self) -> dict: ...
    def generate_mailbox(self, prefix: str | None, domain: str | None) -> dict: ...
    def list_messages(self, email_addr: str) -> list[dict]: ...
    def get_message_detail(self, email_addr: str, message_id: str) -> dict | None: ...
    def delete_message(self, email_addr: str, message_id: str) -> bool: ...
    def clear_messages(self, email_addr: str) -> bool: ...
```

说明：

1. `delete_mailbox` 对上游不是强制能力
2. 用户侧“删除邮箱”允许只做本地删除
3. 如果上游支持删除邮箱，可在 provider 内实现增强

## 4.2 决策二：读取链路统一按“邮箱描述对象”工作，而不是只认 accounts 表

### 结论

现有 `external_api.require_account()` 不能继续作为唯一读取入口。

必须新增统一邮箱解析层，把读取对象抽象成 mailbox descriptor：

```json
{
  "kind": "account | temp",
  "email": "demo@example.com",
  "source": "outlook | imap | custom_temp",
  "status": "active",
  "read_capability": "graph | imap | temp_provider",
  "meta": {}
}
```

### 原因

当前 `/api/external/messages*` 的核心问题不是接口路径，而是服务层只支持：

1. `accounts` 表
2. Outlook/IMAP 普通邮箱

任务邮箱接进来之后，如果不改这里，就算 `apply` 成功，后续外部读信接口也读不到。

### 方案

新增：

1. `mailbox_resolver.resolve_mailbox(email_addr)`
2. `mailbox_resolver.ensure_mailbox_can_read(mailbox_descriptor)`

读取流程改为：

```text
external controller
  -> mailbox_resolver.resolve_mailbox(email)
  -> if kind == account: 走现有 graph / imap 流程
  -> if kind == temp: 走 temp_mail_service 读取流程
```

### 影响点

需要修改：

1. `outlook_web/services/external_api.py`
2. `outlook_web/controllers/emails.py`
3. `outlook_web/security/external_api_guard.py`（如 wait/probe 终止逻辑需要扩展）

## 4.3 决策三：任务邮箱继续复用 `/api/external/messages*`，但访问控制不能只靠 allowed_emails

### 结论

动态任务邮箱不能只依赖 API Key 的 `allowed_emails` 静态名单。

必须为任务邮箱增加“归属当前 API consumer”的授权逻辑。

### 原因

当前 `ensure_external_email_access(email_addr)` 的逻辑是：

1. 读取当前 API Key 的 `allowed_emails`
2. 判断目标邮箱是否在白名单里

这套逻辑对动态创建的任务邮箱会失效，因为：

1. 任务邮箱是运行时生成的
2. 它不可能预先存在于静态白名单

### 技术方案

任务邮箱申请时记录：

1. `consumer_key`
2. `consumer_id`
3. `caller_id`
4. `task_id`

外部读取鉴权时，规则改为：

1. 如果是普通邮箱，继续走原 `allowed_emails` 校验
2. 如果是任务邮箱，允许“创建它的同一个 API consumer”读取
3. 如果任务邮箱已 `finished`，默认拒绝继续读取

### 结论性规则

任务邮箱的外部读取授权以“邮箱归属”优先，不以 `allowed_emails` 静态列表为唯一条件。

## 4.4 决策四：用户侧和任务侧共用同一套提取能力

### 结论

验证码和验证链接提取继续复用现有 extractor 能力，不重新发明一套解析逻辑。

### 统一路径

```text
获取最新邮件摘要
  -> 获取邮件详情
    -> 构造统一 email_obj
      -> verification_extractor
```

### 技术要求

用户侧新增：

1. `GET /api/temp-emails/{email_addr}/extract-verification`

外部侧继续复用：

1. `GET /api/external/verification-code`
2. `GET /api/external/verification-link`

### 结果格式

必须保持以下字段语义一致：

1. `verification_code`
2. `verification_link`
3. `formatted`
4. `matched_email_id`
5. `from`
6. `subject`
7. `received_at`

```text
    -> temp_emails controller

```

## 5.2 外部侧调用链

```text

/api/external/messages*
  -> emails controller
    -> external_api service
      -> mailbox_resolver.resolve_mailbox()

```

## 5.3 建议新增控制器

1. `outlook_web/controllers/external_temp_emails.py`

原因：

1. 不应把任务邮箱申请/结束强塞进 `emails.py`

3. 这样职责最清晰

## 6. 数据模型设计

## 6.1 temp_emails 表改造

当前表结构过于简单：

```sql
id, email, status, created_at, updated_at
```

建议扩展为：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | INTEGER | 是 | 主键 |
| `email` | TEXT | 是 | 完整邮箱地址，唯一 |
| `status` | TEXT | 是 | `active` / `finished` / `deleted` |
| `mailbox_type` | TEXT | 是 | `user` / `task` |
| `visible_in_ui` | INTEGER | 是 | `1/0` |
| `source` | TEXT | 是 | 如 `custom_domain_temp` |
| `prefix` | TEXT | 否 | 邮箱前缀 |
| `domain` | TEXT | 否 | 邮箱域名 |
| `task_token` | TEXT | 否 | 任务令牌，任务邮箱唯一 |
| `consumer_key` | TEXT | 否 | 创建该任务邮箱的外部 API consumer |
| `caller_id` | TEXT | 否 | 调用方标识 |
| `task_id` | TEXT | 否 | 任务标识 |
| `finished_at` | TIMESTAMP | 否 | 结束使用时间 |
| `meta_json` | TEXT | 否 | 扩展元信息 |
| `created_at` | TIMESTAMP | 是 | 创建时间 |
| `updated_at` | TIMESTAMP | 是 | 更新时间 |

### 关键说明

1. `mailbox_type` 是核心字段，比 `created_by` 更适合做持久化主字段
2. `created_by` 可以在 API 输出层由 `mailbox_type` 映射得到
3. 当前系统不是多用户 SaaS，本期不需要 `user_id`
4. `consumer_key` 是任务邮箱授权复用 external 读取接口的关键字段

## 6.2 temp_email_messages 表改造

当前表的最大技术风险是：

1. `message_id` 被做成了全局唯一
2. 这要求上游 message_id 在全系统范围绝对唯一
3. 这个假设不可靠

### 建议目标结构

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | INTEGER | 是 | 主键 |
| `email_address` | TEXT | 是 | 所属邮箱 |
| `message_id` | TEXT | 是 | 上游邮件 ID |
| `source` | TEXT | 是 | 来源标识 |
| `from_address` | TEXT | 否 | 发件人 |
| `subject` | TEXT | 否 | 主题 |
| `content` | TEXT | 否 | 文本正文 |
| `html_content` | TEXT | 否 | HTML 正文 |
| `has_html` | INTEGER | 是 | 是否有 HTML |
| `timestamp` | INTEGER | 否 | 时间戳 |
| `raw_content` | TEXT | 否 | 原始载荷 |
| `created_at` | TIMESTAMP | 是 | 写入时间 |

### 唯一约束建议

改成：

```sql
UNIQUE(email_address, message_id)
```

而不是：

```sql
UNIQUE(message_id)
```

### 原因

1. 不同邮箱下出现相同 message_id 的风险不能忽略
2. 未来多 provider 时，全局唯一更不可靠

### 兼容策略

2. 新 UI 不再展示为正式配置项

## 8. 关键业务流程的技术实现

## 8.1 用户创建邮箱

```text
前端提交 prefix/domain
  -> temp_emails controller
    -> temp_mail_service.generate_user_mailbox()
      -> provider.generate_mailbox()
      -> repo.save(mailbox_type='user', visible_in_ui=1)
      -> 返回 mailbox dto
```

### 要点

1. `prefix` 和 `domain` 在 controller 层做基础校验
2. domain 合法性以配置域名白名单为准
3. 本地 DB 保存的是产品视角状态，不是上游唯一真相

## 8.2 任务申请邮箱

```text
external apply
  -> api_key_required
  -> external guard
  -> temp_mail_service.apply_task_mailbox()
    -> provider.generate_mailbox()
    -> 生成 task_token
    -> repo.save(mailbox_type='task', visible_in_ui=0, consumer_key=当前 consumer)
    -> 返回 email + task_token
```

### task_token 设计要求

1. 使用不可预测随机值
2. 对任务邮箱唯一
3. 不直接使用邮箱地址或 task_id 拼接

## 8.3 用户侧查看邮件

```text
GET /api/temp-emails/{email_addr}/messages
  -> temp_mail_service.sync_messages(email)
    -> provider.list_messages(email)
    -> repo.upsert message cache
    -> 返回格式化列表
```

### 设计说明

1. `temp_email_messages` 是读穿缓存，不是权威真相
2. 刷新时优先回源，再写本地缓存
3. 详情页在缓存不完整时允许补抓详情

## 8.4 外部侧读取任务邮箱

```text
GET /api/external/messages?email=xxx
  -> ensure external auth
  -> mailbox_resolver.resolve_mailbox(email)
  -> if temp mailbox:
       检查 consumer_key 是否匹配
       检查 status 是否 active
       走 temp_mail_service.read_messages_for_external()
```

### 关键点

1. 这里必须同时支持普通邮箱和任务邮箱
2. 不能把 temp mailbox 映射伪装成 account
3. 否则会污染普通邮箱 Graph/IMAP 逻辑

## 8.5 结束任务邮箱使用

```text
POST /api/external/temp-emails/{task_token}/finish
  -> 查 task_token
  -> 校验 consumer_key
  -> status = finished
  -> finished_at = now
```

### 本期技术约束

1. `finish` 只做软结束，不要求立即物理删除
2. `finish` 后默认不再允许 external 读取
3. 后续是否做物理清理，由维护任务再处理

## 8.6 finish 与 wait / probe 的技术行为

由于 external 侧已经存在：

1. `GET /api/external/wait-message`
2. `GET /api/external/probe/{probe_id}`

因此 `finish` 不能只改状态，还要定义等待中的行为。

建议技术规则：

1. `finish` 后，新发起的 `messages*`、`verification-*`、`wait-message` 请求统一返回 `TASK_FINISHED`
2. `finish` 发生时，若存在进行中的同步 `wait-message`，应尽快中断，而不是继续阻塞到 timeout
3. `finish` 发生时，若存在异步 probe，应把 probe 状态改为 `cancelled`
4. `GET /api/external/probe/{probe_id}` 返回 `cancelled` 时，错误码建议固定为 `PROBE_CANCELLED`
5. probe 记录里建议保留 `cancelled_at` 或复用 `updated_at` 作为终止时间

## 9. 前端改造要点

## 9.1 创建区域

需要新增：

3. “自动分配”默认选项

依赖接口：

## 9.2 验证码提取入口

当前错误路径：

```text
copyVerificationInfo -> /api/emails/{email}/extract-verification
```

目标路径：

```text
temp email action -> /api/temp-emails/{email_addr}/extract-verification
```

### 原因

1. 普通邮箱提取接口依赖 `accounts` 表
2. temp mailbox 不应该再走普通邮箱业务分支

### 真实集成点补充

当前前端不是 temp-email 页面自己直接请求提取接口，而是复用了共享函数。

这意味着必须修改的不是一处文案，而是共享调用链：

```text
temp_emails.js 按钮
  -> copyVerificationInfo(email, button)
    -> source-aware 分流
      -> 普通邮箱: /api/emails/{email}/extract-verification
      -> 临时邮箱: /api/temp-emails/{email_addr}/extract-verification
```

建议：

1. 给共享函数显式传入 mailbox source 或当前页面上下文
2. 不要依赖“邮箱是否长得像临时邮箱”这种脆弱判断
3. temp-email 页面需要有独立的 source 标识，而不是只改按钮文案

## 9.3 文案与状态标记

需要去掉：

1. `GPTMail`
2. `currentMethod = 'gptmail'`
3. 以 GPTMail 命名的 UI 标签

建议改成：

1. `temp_mail`
2. `临时邮箱`
3. `自建临时邮箱`

2. 迁移原数据
3. 删除旧表

## 10.2 配置迁移

1. 增加新 `temp_mail_*` 设置项
2. 后台设置页面读取新字段
3. 旧 `gptmail_api_key` 保留一版兼容期
4. 新功能逻辑不再主动依赖旧字段

## 10.3 灰度切换顺序

建议顺序：

1. 先上 schema 与 service abstraction
2. 再切用户侧 temp-emails
3. 再改 external 读取兼容
4. 最后开放 `apply` / `finish`

原因：

如果先开放 `apply`，但 external 读取还不支持 temp mailbox，会形成半成品能力。

## 11.3 上游失败处理

策略：

1. 上游创建失败直接返回，不写本地记录
2. 上游读信失败保留本地缓存，但接口返回失败
3. 上游清空失败时，不默认只删本地，避免真假状态不一致

## 13. 风险与缓解

## 13.1 最大风险

### 风险 A：external 读取改造误伤普通邮箱

缓解：

1. 通过 mailbox resolver 显式分流
2. 不在原有 graph/imap 分支里塞 temp mailbox 特判

### 风险 B：动态任务邮箱绕过邮箱范围控制

缓解：

1. 任务邮箱必须写入 `consumer_key`
2. external 读取时校验当前 consumer 与创建者一致

### 风险 C：message_id 全局唯一导致缓存冲突

缓解：

1. 改成 `(email_address, message_id)` 复合唯一

### 风险 D：用户侧和外部侧解析结果不一致

缓解：

1. 强制共用同一个 extractor service
2. 不允许用户侧单独实现另一套正则提取

## 14. 最终技术结论

1. 这次改造的核心不是“换个 API 地址”，而是把临时邮箱从 provider 直连模式升级为正式能力模块
2. 真正的关键改造点在 service abstraction 和 mailbox resolver，而不是路由数量
3. 任务侧可以复用现有 `/api/external/messages*`，前提是完成读取服务层的统一化
4. `temp_emails` 表必须升级为可表达邮箱类型、可见性和外部归属的模型
5. 本期不需要复杂生命周期，但必须把 `active` / `finished` 这个最小状态闭环做完整
