## 0. 当前实施状态

本 TD 中定义的核心技术收口已完成：

1. `outlook_web.repositories.temp_emails` 已提供 descriptor/public/record 三种视图，并固化 `meta_json` 序列化、反序列化与 legacy 兼容映射。
2. `TempMailProviderBase` 已升级为 mailbox-first contract，`CustomTempMailProvider` 与 factory 已按新 contract 适配。
3. `TempMailService` 已完成 mailbox-first 重构，创建、读取、删除、清理、提取均以 mailbox descriptor 驱动。
4. `mailbox_resolver` 与 `external_api` 已统一通过 descriptor + service 分流 temp mailbox 与普通邮箱读取链路。
5. settings/runtime/provider 收口逻辑已经落地到 `repositories/settings.py` 与 `controllers/settings.py`。

当前技术风险已从“边界未成型”降为“后续多 provider 扩展时的细节一致性硬化”，不再是本期阻塞项。

## 1. 文档目标

本文档只讨论二期 temp-mail 平台化的技术实现，不重复解释产品需求。

本 TD 要解决的是：

1. 如何把当前“已可用的一期 temp-mail”继续升级成内部稳定适配平台。
2. 如何在不改 `/api/temp-emails/*`、`/api/external/*`、`/api/settings` 正式路径的前提下完成内部重构。
3. 如何让 mailbox 成为正式内部资源对象，而不再只是 `email_addr`。
4. 如何让 provider 选择从“默认单实现”升级为“runtime 可切换”。
5. 如何保证 external 兼容链路、settings 契约、verification 语义在重构后不回归。

一句话：

二期不是继续补功能按钮，而是收口内部技术边界。

## 2. 当前实现现状

基于当前代码，系统已经完成了一期产品化，但仍保留明显的二期缺口。

### 2.1 已有基础

1. `TempMailService` 已存在，但当前默认直接绑定 `CustomTempMailProvider()`。
2. `TempMailProviderBase` 已存在，但接口仍以 `email_addr` 为中心。
3. `CustomTempMailProvider` 已存在，但仍桥接 `gptmail.py`。
4. `mailbox_resolver` 已存在，但输出仍偏“分类判断”而不是完整 mailbox descriptor。
5. `temp_emails` 表已经具备 `mailbox_type / visible_in_ui / task_token / consumer_key / meta_json` 等字段。
6. `/api/external/messages*` 已能兼容 temp mailbox。

### 2.2 当前技术缺口

1. provider contract 仍是 email-first，不是 mailbox-first。
2. service 构造方式仍是单 provider 默认绑定，不支持工厂化选择。
3. `meta_json` 虽已存在，但还没有明确 schema 和 handle 语义。
4. settings 与 runtime provider 选择规则尚未完全显式化。
5. custom provider 仍是 bridge provider，而不是正式独立适配层实现。
6. 用户可见 provider 语义和 runtime 真正 provider 类型还有混用风险。

## 3. 二期设计原则

## 3.1 上层 API 稳定，内部边界重构

不改的东西：

1. `/api/temp-emails/*`
2. `POST /api/external/temp-emails/apply`
3. `POST /api/external/temp-emails/{task_token}/finish`
4. `/api/external/messages*`
5. `/api/external/verification-*`
6. `/api/external/wait-message`
7. `/api/settings`

要重构的东西：

1. provider base contract
2. provider factory
3. mailbox descriptor
4. service 调用链
5. resolver 输出结构
6. settings/runtime 选择逻辑

## 3.2 mailbox 是正式资源，不是字符串

从二期开始，temp mailbox 不能只理解成邮箱地址。

任何 temp-mail 相关操作都必须围绕同一个内部对象工作，至少包括：

1. 邮箱地址
2. 类型
3. 状态
4. 可见性
5. 任务归属
6. provider handle
7. provider 类型

## 3.3 provider-specific 信息只允许留在内部

允许存在：

1. provider mailbox id
2. provider token / auth handle
3. provider raw payload
4. provider delete / cursor / polling meta

但这些字段只能存在于：

1. `meta_json`
2. service 内部上下文
3. provider 调用链

不允许直接出现在：

1. 用户侧 API 响应
2. external API 正式字段
3. 前端页面状态

## 4. 目标架构

## 4.1 分层结构

目标调用链：

```text
routes
  -> controllers
    -> service facade
      -> mailbox resolver / repo
      -> provider factory
      -> temp mail provider
      -> extractor
```

其中：

1. controller 只做请求解析、错误映射、响应整形
2. service 负责业务语义和流程编排
3. provider factory 负责 runtime provider 选择
4. provider 负责底层 source 适配
5. repo 负责 mailbox/message 本地状态

## 4.2 模块划分

建议的目标模块如下：

1. `outlook_web/services/temp_mail_provider_base.py`
2. `outlook_web/services/temp_mail_provider_factory.py`
3. `outlook_web/services/temp_mail_provider_custom.py`
4. `outlook_web/services/temp_mail_service.py`
5. `outlook_web/services/mailbox_resolver.py`
6. `outlook_web/repositories/temp_emails.py`

其中：

1. `gptmail.py` 继续保留兼容 bridge，但不再作为正式产品入口
2. controller 不应再直接 import `gptmail.py`

## 5. Mailbox Descriptor 设计

## 5.1 目标结构

建议统一内部 mailbox descriptor：

```json
{
  "kind": "temp",
  "email": "demo123@mail.example.com",
  "source": "custom_domain_temp_mail",
  "provider_name": "custom_domain_temp_mail",
  "mailbox_type": "user",
  "visible_in_ui": true,
  "status": "active",
  "prefix": "demo123",
  "domain": "mail.example.com",
  "task_token": "",
  "consumer_key": "",
  "caller_id": "",
  "task_id": "",
  "read_capability": "temp_provider",
  "meta": {}
}
```

普通邮箱 descriptor 保持统一接口：

```json
{
  "kind": "account",
  "email": "user@outlook.com",
  "source": "graph",
  "provider_name": "outlook_graph",
  "status": "active",
  "read_capability": "graph",
  "meta": {
    "account": {}
  }
}
```

## 5.2 设计要点

1. `kind` 决定走 account 还是 temp 分支。
2. `provider_name` 用于 provider factory 选择，不等同于前端展示用语。
3. `meta` 是唯一允许放 provider-specific handle 的字段。
4. temp mailbox 的 `mailbox_type` 决定用户侧/任务侧授权和 UI 可见性。

## 5.3 repo 返回要求

`temp_emails_repo` 不应只返回简化 DTO，而应支持返回可直接给 service 使用的完整 record。

建议保留两层返回语义：

1. raw record: 面向 service/factory/provider
2. public dto: 面向 controller/response

这样可以避免 service 每次再逆向补字段。

## 6. 数据模型设计

## 6.1 `temp_emails`

当前表已具备二期核心字段，但仍需要正式定义语义。

建议字段语义如下：

| 字段 | 说明 |
|------|------|
| `email` | 产品层唯一邮箱地址 |
| `status` | `active / finished / deleted / errored`，本期至少保证 `active / finished` |
| `mailbox_type` | `user / task` |
| `visible_in_ui` | 用户侧列表是否可见 |
| `source` | 产品层来源标识，建议维持 `custom_domain_temp_mail` |
| `prefix` | 邮箱前缀 |
| `domain` | 邮箱域名 |
| `task_token` | 任务邮箱唯一令牌 |
| `consumer_key` | 创建该任务邮箱的 external consumer |
| `caller_id` | 调用方标识 |
| `task_id` | 调用方任务标识 |
| `finished_at` | finish 时间 |
| `meta_json` | provider handle 与 provider runtime 附加信息 |

## 6.2 `meta_json` schema

建议把 `meta_json` 视为正式扩展位，并约定最小结构：

```json
{
  "provider_mailbox_id": "",
  "provider_cursor": "",
  "provider_labels": [],
  "provider_capabilities": {
    "delete_mailbox": false,
    "delete_message": true,
    "clear_messages": true
  },
  "provider_debug": {
    "bridge": "gptmail"
  }
}
```

说明：

1. 本期不要求所有 provider 都填满这些字段。
2. 但新 provider 接入时，只允许往这个结构里扩展，不允许把字段散落到响应层。
3. `provider_debug` 仅用于内部调试，不应对外输出。

## 6.3 `temp_email_messages`

本表当前已经按 `(email_address, message_id)` 作用域工作，这一设计在二期继续保留。

建议继续明确：

| 字段 | 说明 |
|------|------|
| `email_address` | 所属邮箱 |
| `message_id` | provider 消息 ID |
| `from_address` | 发件人 |
| `subject` | 主题 |
| `content` | 文本正文 |
| `html_content` | HTML 正文 |
| `has_html` | 是否有 HTML |
| `timestamp` | provider 返回时间戳 |
| `raw_content` | provider raw payload |

说明：

1. `raw_content` 是本地缓存，不是对外契约。
2. service/detail/extractor 优先消费统一整形后的字段。
3. provider raw payload 只用于补抓详情和 richer payload 选择。

## 7. Provider Contract 设计

## 7.1 当前问题

当前 `TempMailProviderBase` 接口是：

1. `generate_mailbox(...)`
2. `list_messages(email_addr)`
3. `get_message_detail(email_addr, message_id)`
4. `delete_message(email_addr, message_id)`
5. `clear_messages(email_addr)`

问题在于：

1. provider 方法不知道 mailbox 的任务归属、类型、meta handle
2. service 只能把 `email_addr` 当作全部上下文
3. 无法支撑多 provider 和 provider handle 扩展

## 7.2 目标接口

建议升级为：

```python
class TempMailProviderBase:
    def get_options(self) -> dict[str, Any]: ...
    def create_mailbox(self, *, prefix: str | None = None, domain: str | None = None) -> dict[str, Any]: ...
    def delete_mailbox(self, mailbox: dict[str, Any]) -> bool: ...
    def list_messages(self, mailbox: dict[str, Any]) -> list[dict[str, Any]] | None: ...
    def get_message_detail(self, mailbox: dict[str, Any], message_id: str) -> dict[str, Any] | None: ...
    def delete_message(self, mailbox: dict[str, Any], message_id: str) -> bool: ...
    def clear_messages(self, mailbox: dict[str, Any]) -> bool: ...
```

## 7.3 兼容迁移策略

为降低改造成本，二期允许短期双层兼容：

1. base contract 新增 mailbox-first 方法
2. custom provider 内部可先适配旧 bridge
3. `generate_mailbox(...)` 可作为兼容 alias 短期保留

但最终 service 只能调用 mailbox-first 版本。

## 8. Provider Factory 设计

## 8.1 目标

新增统一工厂：

```python
def get_temp_mail_provider(provider_name: str | None = None) -> TempMailProviderBase: ...
```

## 8.2 选择规则

provider 选择优先级建议：

1. 显式传入的 `provider_name`
2. `settings.temp_mail_provider`
3. legacy fallback

其中 legacy fallback 只用于兼容，不得覆盖显式设置。

## 8.3 当前建议 provider 列表

本期至少支持：

1. `custom_domain_temp_mail`
2. `legacy_bridge`

含义：

1. `custom_domain_temp_mail` 是正式 temp-mail provider 语义
2. `legacy_bridge` 是内部桥接语义，不允许对前端展示

## 8.4 工厂错误处理

建议新增错误：

1. `TEMP_MAIL_PROVIDER_NOT_CONFIGURED`
2. `TEMP_MAIL_PROVIDER_INVALID`
3. `TEMP_MAIL_PROVIDER_UNAVAILABLE`

用于区分：

1. 没配 provider
2. provider 值非法
3. provider 当前不可用

## 9. Custom Provider 设计

## 9.1 当前状态

`CustomTempMailProvider` 当前仍桥接 `gptmail.py`，主要做了两件事：

1. 读取 `temp_mail_*` settings
2. 把 bridge 错误映射成统一错误码

## 9.2 二期目标

二期不要求立即消灭 bridge，但要求把 bridge 限定在 provider 内部：

```text
TempMailService
  -> provider factory
    -> CustomTempMailProvider
      -> gptmail bridge
```

不允许出现：

```text
controller -> gptmail
external_api -> gptmail
frontend -> provider url
```

## 9.3 `get_options()` 语义

`CustomTempMailProvider.get_options()` 应负责输出统一 options：

1. `domain_strategy`
2. `default_mode`
3. `domains`
4. `prefix_rules`
5. `provider`
6. `api_base_url`

建议后续将用户可见 provider 语义与 runtime provider 语义拆开，例如：

1. `provider_label`: `temp_mail`
2. `provider_name`: `custom_domain_temp_mail`

避免当前 `"provider": "temp_mail"` 和 `temp_mail_provider=custom_domain_temp_mail` 之间的混用。

## 9.4 create_mailbox 返回语义

建议 create 返回结构最少包含：

```json
{
  "success": true,
  "email": "demo123@mail.example.com",
  "meta": {}
}
```

说明：

1. provider 可以返回 `meta`
2. service 负责把 `meta` 落到 `meta_json`
3. controller 不应直接回显 `meta`

## 10. TempMailService 设计

## 10.1 目标职责

`TempMailService` 在二期应成为业务语义层，而不是 provider 直通层。

职责应包括：

1. prefix/domain 校验
2. mailbox record 读写
3. provider 选择
4. provider 调用编排
5. message/detail DTO 整形
6. verification extractor 调用
7. 任务邮箱 apply / finish 生命周期

## 10.2 目标调用链

建议所有 temp mailbox 读写流程统一为：

```text
service entry
  -> load mailbox record
  -> resolve provider name
  -> get provider from factory
  -> call provider(mailbox, ...)
  -> persist cache / state
  -> return public dto
```

## 10.3 创建流程

### 用户侧

```text
generate_user_mailbox()
  -> validate prefix/domain
  -> provider = factory.get(...)
  -> provider.create_mailbox(...)
  -> repo.create_temp_email(..., mailbox_type=user, visible_in_ui=1, meta_json=provider meta)
  -> return public mailbox dto
```

### 任务侧

```text
apply_task_mailbox()
  -> validate caller_id/task_id/prefix/domain
  -> provider = factory.get(...)
  -> provider.create_mailbox(...)
  -> generate task_token
  -> repo.create_temp_email(..., mailbox_type=task, visible_in_ui=0, consumer_key=...)
  -> return public mailbox dto
```

## 10.4 读取流程

```text
list_messages(email)
  -> mailbox = repo.get_temp_email_by_address(email)
  -> provider = factory.get(mailbox.provider_name)
  -> provider.list_messages(mailbox)
  -> repo.save_temp_email_messages(...)
  -> return summaries
```

detail/delete/clear 同理。

## 10.5 删除/清理原则

必须统一以下规则：

1. provider 明确失败时，不删本地缓存
2. provider 成功后，再删本地状态
3. 若 provider 不支持 `delete_mailbox`，邮箱删除可以仅做本地删除，但要通过 service 统一封装

## 10.6 verification 提取

用户侧和 external 侧继续共用统一 extractor：

```text
latest message summary
  -> detail
  -> extract_verification_info_with_options(...)
  -> normalize result
```

返回字段至少保持：

1. `verification_code`
2. `verification_link`
3. `formatted`
4. `matched_email_id`
5. `from`
6. `subject`
7. `received_at`

## 11. mailbox_resolver 与 external 设计

## 11.1 当前问题

当前 `mailbox_resolver` 已能区分 `account` 和 `temp`，但输出还偏轻量：

1. temp 分支主要返回 `meta.temp_mailbox`
2. account 分支主要返回 `meta.account`
3. service/factory 还没有围绕统一 descriptor 工作

## 11.2 二期目标

resolver 输出应成为 external 侧统一入口对象。

建议能力：

1. `resolve_mailbox(email_addr)` 返回统一 descriptor
2. `ensure_mailbox_can_read(mailbox, consumer, allow_finished=False)` 负责读权限
3. `ensure_mailbox_can_mutate(mailbox, consumer)` 预留给后续写操作

## 11.3 external/messages* 调用链

目标调用链：

```text
emails controller
  -> external_api service
    -> mailbox_resolver.resolve_mailbox(email)
    -> mailbox_resolver.ensure_mailbox_can_read(...)
    -> if account: graph/imap
    -> if temp: temp_mail_service
```

重点：

1. 普通邮箱逻辑不被 temp mailbox 特判污染
2. temp mailbox 不伪装成 account
3. external 侧只认识 mailbox descriptor，不直接碰 provider

## 11.4 finish / wait / probe

二期必须统一这些语义：

1. finish 后新的 external 读请求返回 `TASK_FINISHED`
2. 正在等待的同步 `wait-message` 应尽快终止
3. 正在运行的异步 `probe` 应转为 `cancelled`
4. 查询 probe 时优先返回 probe 终态，不应再被邮箱可读性覆盖

## 12. settings 与 runtime 设计

## 12.1 正式字段

二期继续以 `/api/settings` 为唯一正式配置入口，核心字段包括：

1. `temp_mail_provider`
2. `temp_mail_api_base_url`
3. `temp_mail_api_key`
4. `temp_mail_api_key_set`
5. `temp_mail_api_key_masked`
6. `temp_mail_domains`
7. `temp_mail_default_domain`
8. `temp_mail_prefix_rules`

## 12.2 目标语义

建议把配置语义拆成两层：

1. 用户/前端看到的是 temp-mail 正式语义
2. runtime 内部消费的是 provider 选择语义

例如：

1. 前端 options 可展示 `provider_label=temp_mail`
2. runtime 仍使用 `provider_name=custom_domain_temp_mail`

## 12.3 写入规则

`api_update_settings()` 应满足：

1. `temp_mail_api_key` 空字符串默认不覆盖已有值
2. masked placeholder 不覆盖明文
3. legacy `gptmail_api_key` 只允许正向迁移，不允许反向清空正式字段
4. 显式设置优先于 env fallback

## 12.4 env fallback

二期允许保留 legacy env fallback，但建议明确规则：

1. 先读 DB settings
2. DB 未配置时才读 env fallback
3. env fallback 只解决兼容，不得作为正式 UI 展示来源

## 13. 迁移策略

## 13.1 代码迁移顺序

建议顺序：

1. 先定义 mailbox descriptor
2. 再升级 provider base
3. 再引入 provider factory
4. 再改 `TempMailService`
5. 再改 `mailbox_resolver` 和 external
6. 最后收口 settings/runtime

原因：

如果先改 external 或 settings，但底层 service/provider 接口还没统一，代码会进入半重构状态。

## 13.2 兼容策略

短期保留：

1. `gptmail.py`
2. legacy `gptmail_api_key`
3. `legacy_gptmail` source

但限制为：

1. 仅内部 bridge 使用
2. 不再作为正式产品字段
3. 不再作为 controller/external 直接入口

## 13.3 风险控制

最大风险有三类：

1. 重构 service/provider 时误伤一期 API
2. external 分流时误伤 Outlook / IMAP
3. settings/runtime 迁移时造成 provider 选择漂移

缓解方式：

1. 先补 contract tests
2. 保留 external compat 回归
3. 保留 settings/platform contract 回归

5. settings/runtime -> `tests/test_temp_mail_settings_platform_contract.py`
6. external compat -> `tests/test_external_api_temp_mail_compat.py`
7. 用户侧 API 回归 -> `tests/test_temp_emails_api_regression.py`
8. 前端契约 -> `tests/test_temp_mail_target_contract.py`

## 15. 最终技术结论

1. 二期真正的技术改造点，不是再加 API 路径，而是让内部 mailbox / provider / service / resolver / settings 五层形成稳定边界。
2. 当前最优解不是立刻物理移除 bridge，而是先把 bridge 严格收口到 provider 内部。
3. `TempMailService` 必须从“默认 provider 直连”升级为“mailbox-first + provider factory”。
4. `mailbox_resolver` 必须从“邮箱分类器”升级为“统一 descriptor 入口”。
5. 只有当内部 provider 可替换、上层 API 不变、external 不回归、settings/runtime 一致时，二期平台化才算真正完成。
